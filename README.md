# NDT Web Application

Веб-приложение для учета неразрушающего контроля сварных соединений.

Система предназначена для цифрового учета заявок, стыков, методов контроля, результатов, заключений, сменных журналов, реестров передачи, архива материалов, оборудования, сотрудников и связанных документов.

Основной проектный документ: `project.md`.

## Локальная разработка

Этап 1 уже подготавливает базовый Laravel-каркас, Docker Compose и конфиги для фронтенд-сборки.

Базовые команды после установки зависимостей:

```bash
composer install
npm install
cp .env.example .env
php artisan key:generate
docker compose -f docker/compose.yml up --build
```

Отдельно доступны:

- `php artisan test`
- `php artisan app:ping`
- `npm run build`

## Назначение

Приложение должно заменить ручной табличный учет и стать единой рабочей системой для участка или лаборатории НК.

Ключевые задачи:

- учет заявок на контроль;
- учет сварных стыков;
- назначение методов НК;
- распределение заданий исполнителям;
- фиксация результатов контроля;
- учет радиографического контроля, пленок, снимков, пересветов и архива;
- учет смен лаборанта и дешифровщика;
- подготовка и утверждение заключений;
- ведение реестров, актов, документов и файлов;
- учет сотрудников, квалификаций и оборудования;
- разграничение доступа по ролям и объектам/участкам.

## Основные принципы

Главные требования к системе:

- простота использования;
- быстродействие;
- простая и понятная БД;

## Архитектура

Тип системы: модульный монолит.

Основные backend-модули:

- `Auth` — вход, пользователи, сессии, API-токены;
- `Access` — роли, права, политики доступа;
- `Employees` — сотрудники, должности, квалификации, удостоверения;
- `Organizations` — организации, заказчики, лаборатория, реквизиты;
- `Objects` — города, объекты/участки, титулы, чертежи, линии;
- `Welds` — сварные соединения, сварщики, материалы, параметры стыков;
- `NdtRequests` — заявки НК;
- `NdtTasks` — задания исполнителям;
- `NdtResults` — результаты контроля по методам;
- `Radiography` — РК, пленки, снимки, пересветы, плотности, архив;
- `VisualControl` — ВИК;
- `PenetrantControl` — ПВК;
- `MagneticControl` — МК;
- `UltrasonicControl` — УК;
- `Conclusions` — заключения, позиции заключений, статусы выдачи;
- `Registers` — реестры передачи, акты, архивные передачи;
- `Shifts` — смены лаборанта и дешифровщика;
- `Inventory` — пленка, химия, расходники;
- `Equipment` — оборудование, поверки, калибровки, ремонты, выдачи;
- `Documents` — документы, файлы, версии, связи;
- `Notifications` — уведомления, очередь отправки;
- `Audit` — журнал действий и изменений;
- `Reports` — отчеты, выгрузки, печатные формы;
- `Admin` — системные справочники и настройки.

## Роли

Базовые роли системы:

- Администратор системы;
- Начальник участка;
- Инженер НК / Дешифровщик;
- Дефектоскопист;
- Лаборант.

Администратор видит все данные и управляет всеми разделами.

Начальник участка работает с данными своего объекта/участка.

Каждый сотрудник закрепляется за конкретным объектом/участком. Объект/участок относится к городу. В одном городе может быть один или несколько объектов/участков.

## Основной процесс

Базовый производственный процесс:

```text
Заявка
→ регистрация
→ проверка полноты данных
→ список стыков
→ назначение методов НК
→ планирование заданий
→ выполнение контроля
→ передача материалов
→ лабораторная обработка / дешифровка
→ анализ результата
→ заключение
→ утверждение
→ выдача
→ архив
```

## Методы контроля

Система должна поддерживать методы НК как отдельные сущности, связанные со стыком через назначение метода.

Базовые методы:

- РК — радиографический контроль;
- ВИК — визуальный и измерительный контроль;
- ПВК — капиллярный контроль;
- МК — магнитопорошковый контроль;
- УК — ультразвуковой контроль.

Один стык может иметь несколько методов контроля.

## Смены

Система должна поддерживать два типа смен:

- смена лаборанта;
- смена дешифровщика.

Один сотрудник не должен иметь две открытые смены одного типа одновременно.

Смена лаборанта нужна для учета операций с пленкой, химией, расходниками и регламентными работами.

Смена дешифровщика нужна для учета просмотра пленок/снимков, фиксации брака, подозрений на подлог, очистки рабочего места и внесения результатов дешифровки.

## База данных

База данных должна быть нормализованной и понятной. Не нужно собирать всю предметную область в одну большую таблицу.

Ключевые группы таблиц:

- пользователи и доступ;
- сотрудники;
- организации и объекты;
- сварные стыки;
- заявки;
- методы и задания;
- результаты контроля;
- РК;
- ВИК / ПВК / МК / УК;
- заключения;
- реестры;
- смены;
- оборудование;
- документы и файлы;
- уведомления;
- справочники.

Для больших списков обязательна серверная пагинация. Фильтры должны выполняться на стороне БД. Под реальные фильтры должны быть созданы индексы.

## Файлы

Файлы должны храниться приватно.

Скачивание файлов должно выполняться только через backend с проверкой прав доступа к связанной сущности.

Для значимых документов требуется хранение версий.

## Качество кода

Обязательные правила:

- `declare(strict_types=1);`;
- типизированные свойства и методы;
- FormRequest для валидации;
- DTO для сложных входных данных;
- Service-классы для бизнес-операций;
- Policies для доступа;
- Enum-классы для статусов и типов;
- PHPDoc для публичных методов сервисов, DTO, сложных моделей и нетривиальных расчетов;
- комментарии по ходу кода только там, где без них непонятна предметная логика;
- миграции с индексами и FK;
- сидеры для базовых справочников;
- тесты для критичных бизнес-сценариев.

Контроллер не должен содержать бизнес-логику. Контроллер принимает запрос, вызывает сервис и возвращает ответ.

## Production-критерии

Система считается готовой к production, когда:

1. Все основные роли заведены и проверены.
2. Администратор видит все данные и управляет всеми разделами.
3. Начальник участка видит и контролирует данные своего объекта/участка.
4. Каждый сотрудник закреплен за объектом/участком.
5. Города и объекты/участки заведены как отдельные сущности.
6. Заявки, стыки, методы, задания, результаты и заключения работают как единый процесс.
7. Смены лаборанта и дешифровщика имеют обязательные журналы и отчеты.
8. РК, ВИК, ПВК, МК и УК поддерживаются как отдельные методы контроля.
9. Реестры, акты и архив связаны с результатами и заключениями.
10. Файлы хранятся приватно и скачиваются только с проверкой прав.
11. Все критичные изменения пишутся в audit log.
12. Работают очереди для тяжелых операций.
13. Настроены backup БД и файлов.
14. Настроены роли, права, политики и блокировка пользователей.
15. Утвержденные заключения защищены от прямого изменения.
16. Списки работают быстро на больших объемах данных.
17. Есть базовые автотесты критичных бизнес-сценариев.
18. Код документирован PHPDoc там, где это нужно для понимания предметной логики.
19. Есть эксплуатационная инструкция для запуска, обновления, backup и восстановления.
