# AGENTS.md

Инструкция для AI-агентов, Codex и других автоматизированных помощников, работающих с этим проектом.

## 1. Назначение проекта

Проект — production-level веб-приложение для учета неразрушающего контроля сварных соединений.

Система должна учитывать:

- заявки НК;
- сварные стыки;
- методы контроля;
- задания исполнителям;
- результаты контроля;
- радиографический контроль;
- пленки, снимки, пересветы и архив;
- смены лаборанта и дешифровщика;
- заключения;
- реестры и акты;
- оборудование;
- сотрудников и квалификации;
- документы и файлы;
- уведомления;
- audit log.

Основной проектный документ: `project.md`.

Перед началом любой существенной задачи нужно прочитать:

1. `README.md`;
2. `docs/project.md`;
3. профильные документы из `docs/`, если они есть.

## 2. Главные правила работы

Не выдумывать сущности, поля, роли, статусы, процессы, API, таблицы, UI-экраны и бизнес-правила.

Если данных недостаточно, нужно явно написать:

```text
Недостаточно данных для точного ответа.
```

И указать, каких именно данных не хватает.

Не расширять задачу без прямой просьбы.

Не добавлять новые технологии без прямой необходимости.

Не менять утвержденные термины без прямой команды.

Не заменять клиентские термины на более привычные технические синонимы.

Важно использовать именно такие термины:

- `Начальник участка`;
- `Инженер НК / Дешифровщик`;
- `Дефектоскопист`;
- `Лаборант`;
- `объект / участок`;
- `город`;
- `заявка`;
- `стык`;
- `заключение`;
- `реестр`;
- `смена лаборанта`;
- `смена дешифровщика`;
- `дешифровка`.

Не использовать термин `расшифровщик`, если задача прямо не просит обсудить терминологию. В проекте используется термин `Дешифровщик`.

## 5. Код

Обязательные правила backend-кода:

- использовать `declare(strict_types=1);`;
- типизировать аргументы, возвращаемые значения и свойства;
- использовать FormRequest для HTTP-валидации;
- использовать DTO для сложных входных данных;
- держать бизнес-логику в сервисах;
- держать контроллеры тонкими;
- использовать policies для доступа;
- использовать enum-классы для статусов и типов;
- не писать бизнес-логику во Blade/Livewire-компонентах;
- не писать SQL без необходимости, если Eloquent/Query Builder достаточно;
- не делать N+1-запросы;
- не загружать большие файлы целиком в память без необходимости.

Контроллер должен:

1. принять запрос;
2. вызвать FormRequest;
3. вызвать сервис;
4. вернуть redirect, view, Livewire response или JSON.

Контроллер не должен принимать бизнес-решения.

## 6. Комментарии и PHPDoc

Код должен быть документирован там, где это помогает сопровождению.

PHPDoc обязателен для:

- публичных методов сервисов;
- DTO;
- сложных моделей;
- нетривиальных расчетов;
- методов, реализующих предметные правила;
- методов, меняющих статусы критичных сущностей;
- методов, работающих с файлами, заключениями, сменами и audit log.

Комментарии по ходу кода нужны только там, где без них непонятна предметная логика.

Не комментировать очевидный код.

Плохой комментарий:

```php
// Создаем пользователя
$user = User::create($data);
```

Хороший комментарий:

```php
// Утвержденное заключение нельзя редактировать напрямую:
 // исправления должны идти через новую версию или аннулирование.
```

## 7. База данных

База данных должна быть простой, нормализованной и понятной.

Не собирать всю систему в одну широкую таблицу.

Обязательные правила миграций:

- задавать FK там, где связь обязательна;
- добавлять индексы под реальные фильтры;
- использовать enum/string-статусы осознанно;
- не удалять используемые справочные значения физически;
- для рабочих сущностей использовать soft delete или статус аннулирования, если данные могут быть связаны с документами;
- не хранить несколько методов контроля строкой через запятую;
- не хранить файлы в БД;
- не хранить производственные результаты без audit trail.

Ключевая объектная модель:

- город;
- объект / участок;
- сотрудник;
- пользователь;
- заявка;
- стык;
- метод контроля;
- задание;
- результат;
- заключение;
- смена;
- файл;
- документ;
- оборудование;
- реестр;
- audit log.

Каждый сотрудник закреплен за объектом/участком.

Объект/участок относится к городу.

В одном городе может быть один или несколько объектов/участков.

## 8. Доступ и роли

Базовые роли:

- Администратор системы;
- Начальник участка;
- Инженер НК / Дешифровщик;
- Дефектоскопист;
- Лаборант.

Администратор:

- видит все данные;
- управляет всеми разделами;
- управляет пользователями, ролями, правами, справочниками, объектами и настройками.

Начальник участка:

- работает с данными своего объекта/участка;
- видит сотрудников своего объекта/участка;
- распределяет задания;
- контролирует заявки, стыки, результаты и заключения по своему объекту/участку.

Инженер НК / Дешифровщик:

- анализирует материалы контроля;
- выполняет дешифровку снимков/пленок;
- фиксирует дефекты, брак, подозрения на подлог;
- оформляет результаты;
- готовит заключения;
- работает в сменном сценарии дешифровки.

Дефектоскопист:

- выполняет контроль;
- фиксирует выполнение задания;
- передает результат дальше.

Лаборант:

- ведет смену лаборанта;
- работает с пленкой, химией, расходниками;
- передает материалы на дешифровку.

Любое действие должно проверяться через policy или gate.

## 9. Объектная область НК

Один стык может иметь несколько методов контроля.

Базовые методы:

- РК;
- ВИК;
- ПВК;
- МК;
- УК.

Метод контроля должен быть отдельной сущностью/связью, а не строковым списком внутри стыка.

РК требует отдельного расширенного контура:

- пленки;
- снимки;
- пересветы;
- плотности;
- брак;
- дешифровка;
- архив;
- реестры передачи.

## 10. Смены

В системе есть смены:

- смена лаборанта;
- смена дешифровщика.

Один сотрудник не должен иметь две открытые смены одного типа одновременно.

Смена лаборанта:

- старт смены;
- прием пленки;
- выдача пленки;
- списание пленки;
- поступление химии;
- замена химии;
- запрос химии;
- регламентные работы;
- передача материалов на дешифровку;
- завершение смены.

Смена дешифровщика:

- старт смены;
- получение списка пленок/снимков;
- фиксация просмотренных групп;
- фиксация брака;
- фиксация подозрения на подлог;
- фиксация очистки рабочего места;
- внесение результатов дешифровки;
- передача результатов на анализ;
- завершение смены.

## 11. Заключения

Утвержденное заключение нельзя редактировать напрямую.

Исправления должны выполняться через:

- новую версию;
- замену;
- аннулирование;
- корректирующую запись.

Любое изменение результата контроля или заключения должно попадать в audit log.

## 12. Файлы

Файлы хранятся приватно.

Скачивание только через backend с проверкой прав.

Использовать:

- Laravel Storage;
- S3-compatible хранилище;
- signed download URL;
- метаданные файлов в БД;
- версии для значимых документов.

Не хранить файлы в публичной директории без проверки прав.

Не отдавать прямые storage-пути пользователю.

## 13. Audit log

Audit log обязателен для:

- заявок;
- стыков;
- методов контроля;
- заданий;
- результатов;
- заключений;
- реестров;
- файлов;
- смен;
- оборудования;
- квалификаций;
- пользователей и прав.

Audit log должен фиксировать:

- кто изменил;
- когда изменил;
- какую сущность изменил;
- старые значения;
- новые значения;
- IP;
- user-agent;
- причину изменения, если она требуется.

## 14. Производительность

Обязательные правила:

- все большие списки с серверной пагинацией;
- фильтры на стороне БД;
- индексы под реальные фильтры;
- тяжелые экспорты через очередь;
- PDF-генерация через очередь;
- загрузка больших файлов потоковая;
- dashboard без тяжелых агрегатов на каждом запросе;
- не допускать N+1;
- eager loading только нужных связей.

API нужен для PWA/mobile-сценариев и возможных интеграций.

Базовые группы:

- `/api/auth`;
- `/api/profile`;
- `/api/mobile/tasks`;
- `/api/mobile/shifts`;
- `/api/mobile/welds`;
- `/api/mobile/files`;

API должен использовать:

- Sanctum;
- rate limiting;
- policies;
- DTO;
- единый формат ошибок;
- audit log для критичных действий.

## 16. UI

UI должен быть рабочим, простым и быстрым.

Не делать декоративную сложность.

Основной интерфейс:

- таблицы;
- фильтры;
- карточки;
- формы;
- модалки;
- журналы;
- статусы;
- действия по ролям.

Для Android/планшетов использовать PWA-экраны.

Нативное Android-приложение не закладывать без отдельного подтверждения необходимости.

Обязательные направления:

- авторизация;
- права доступа;
- создание города;
- создание объекта/участка;
- закрепление сотрудника за объектом/участком;
- ограничение начальника участка данными своего объекта/участка;
- полный доступ администратора ко всем объектам/участкам;
- создание заявки;
- добавление стыков;
- назначение методов;
- назначение задания;
- выполнение задания;
- создание результата;
- результат с дефектом;
- пересвет;
- формирование заключения;
- утверждение заключения;
- запрет редактирования утвержденного заключения;

## 18. Работа с задачами

При выполнении задачи агент должен:

1. Понять, к какому модулю относится задача.
2. Прочитать существующий код модуля.
3. Проверить проектную документацию.
4. Внести минимально необходимые изменения.
5. Не менять соседние области без причины.
6. Добавить или обновить тесты, если меняется бизнес-логика.
7. Обновить документацию, если меняется архитектура, БД, API, роли или бизнес-процесс.

Если задача неоднозначна, не придумывать недостающие правила.

## 19. Запрещенные действия без прямой команды

Не делать без прямой команды:

- менять утвержденный стек;
- добавлять Vue как обязательный frontend;
- добавлять микросервисы;
- добавлять GraphQL;
- добавлять Elasticsearch;
- менять названия ролей;
- менять термин `Дешифровщик`;
- объединять методы контроля в строковое поле;
- делать публичную выдачу файлов без проверки прав;
- редактировать утвержденные заключения напрямую;
- удалять производственные данные физически;
- отключать audit log;
- убирать проверку object/участок scope для начальника участка;
- добавлять роли, которых нет в проектном документе;
- проектировать пользовательские сценарии, которых нет в документации или задаче.

## 20. Формат ответа агента

Если агент пишет план, он должен быть конкретным:

- какие файлы будут изменены;
- какие таблицы будут созданы или изменены;
- какие сервисы появятся;
- какие FormRequest/DTO/Policy/Enum нужны;
- какие тесты будут добавлены;
- какие риски или недостающие данные есть.

Если агент пишет код, он не должен дублировать в ответе весь код созданных файлов, если файлы уже созданы в репозитории. Достаточно краткого перечня изменений и путей файлов.

Если агент не может выполнить задачу точно, он должен написать, чего именно не хватает.
