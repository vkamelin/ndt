# Анализ документации веб-приложения конкурента

## 1. Назначение документа

Документ подготовлен как аналитический разбор PDF-файла `Lab365 — User Guide.pdf` как самостоятельного источника. Цель — выделить из руководства пользователя структуру веб-приложения, модули, сущности, поля, роли, статусы, процессы, документы, справочники, ограничения и возможные проектные требования для собственного веб-приложения.

Анализ не является копированием продукта Lab365 и не содержит дословного воспроизведения экранов, шаблонов документов или пользовательских формулировок. PDF используется только как источник для выявления функциональных зон, предметной логики и возможных архитектурных ориентиров.

## 3. Краткое резюме анализа

Lab365 описана как LIMS-платформа для испытательных лабораторий в промышленности. Система ориентирована на управление лабораторной информацией, оборудованием, метрологической прослеживаемостью, специалистами, документами, аттестационным комплектом лаборатории, объектами испытаний и заключениями результатов испытаний.

Основной акцент PDF сделан не на оперативном производственном цикле «заявка → задание → контроль → результат», а на системообразующих данных лаборатории: организация, лаборатория, пользователи, роли, оборудование, поверки/калибровки, специалисты, удостоверения, документы, справочники и формирование комплекта документов для аттестации лаборатории НК.

Из PDF явно видны следующие крупные зоны системы:

| Зона | Что подтверждено PDF |
|---|---|
| Доступ и аккаунты | регистрация организации, активация аккаунта, вход, восстановление пароля, профиль пользователя |
| Организация и лаборатория | карточка организации, реквизиты, данные лаборатории, ответственные лица, документы организации и лаборатории |
| Пользователи и роли | ограниченный набор ролей, приглашения, блокировка, разблокировка, смена роли |
| Уведомления | системные уведомления, список уведомлений, счетчик, email-уведомления, уведомления о сроках поверки/калибровки и удостоверений |
| Склад | учет оборудования, моделей, типов, состояний, поверок/калибровок, ремонтов, материалов, формирование документов по оборудованию |
| Специалисты | учет сотрудников лаборатории, статусов, должностей, удостоверений НК/ПБ, допусков, спецодежды, документов |
| Документы | хранение документов, формирование комплекта документов для аттестации лаборатории, скачивание документов и архивов |
| Объекты и заключения | модуль заявлен, но детально не раскрыт в предоставленном PDF |

Главное ограничение: PDF не содержит описания БД, API, технической архитектуры, внутренней схемы прав, журналирования, бизнес-правил объектов и заключений, а также полного жизненного цикла испытаний. Все проектные выводы по этим зонам должны считаться предположениями или вопросами к уточнению.

## 5. Назначение веб-приложения

Lab365 предназначена для цифровизации работы испытательных лабораторий и управления результатами испытаний. По структуре PDF видно, что система решает не только задачу оформления итоговых заключений, но и задачу поддержания готовности лаборатории: специалисты должны иметь актуальные удостоверения, оборудование должно иметь актуальные поверки/калибровки, документы лаборатории должны храниться и переиспользоваться, а комплект для аттестации должен формироваться на основе уже введенных данных.

| Аспект | Вывод |
|---|---|
| Основная предметная область | Испытательные лаборатории, промышленный контроль, лаборатории НК. |
| Ключевая задача | Управлять лабораторными данными, оборудованием, специалистами, документами и результатами испытаний. |
| Автоматизируемые процессы | Регистрация организации, управление пользователями, учет оборудования, контроль поверок/калибровок, учет специалистов и удостоверений, хранение документов, формирование аттестационного комплекта, работа с объектами и заключениями. |
| Пользователи системы | Владелец организации, начальник лаборатории, инженер-дефектоскопист, дефектоскопист. Возможно участие корпоративных структур и подрядных лабораторий при внедрении, но детали не раскрыты. |
| Хранимые данные | Данные организации, лаборатории, пользователей, специалистов, оборудования, документов, поверок, калибровок, ремонтов, удостоверений, допусков, нормативно-технической документации, объектов и заключений. |
| Формируемые документы | Комплект документов для аттестации лаборатории, графики поверки и аттестации, списки оборудования и специалистов, учетные карточки, заявки на поверку, журналы заключений, заключения результатов испытаний. |
| Бизнес-проблемы | Снижение ручного учета, контроль сроков действия документов и поверок, автоматизация комплектов документов, централизованное хранение данных лаборатории, ограничение доступа по ролям. |

Если восстанавливать назначение системы по структуре PDF, Lab365 — это не только система оформления результатов НК. Это учетно-документальная LIMS-система, в которой операционный модуль объектов и заключений зависит от предварительно заполненных системообразующих модулей: профиль организации, лаборатория, склад, специалисты и документы.

## 10. Роли пользователей и права доступа

### 11.11 Работа с объектами и заключениями

PDF заявляет, что модуль предназначен для планирования и управления ходом испытаний, хранения данных испытаний и формирования рабочей документации. По оглавлению видны действия: добавить объект, список объектов, поиск и фильтрация, журнал заключений, карточка объекта, технические устройства, типоразмеры, элементы контроля, специалисты и оборудование на объекте, добавление заключения, список заключений, скачивание и удаление заключений.

Недостаточно данных для точного вывода по этапам процесса, ролям, статусам, обязательным полям и жизненному циклу заключения. Эти сведения отсутствуют в предоставленной части PDF.

| Модель оборудования | поиск в каталоге → выбор или создание новой → модерация → появление в каталоге | не описан процесс модерации и роли модератора |
| Специалист | добавление → ведение документов и удостоверений → изменение статуса → исключение через статус «уволен» | нет удаления, нет истории трудовых изменений |
| Удостоверение | добавление → актуальное действие → истечение → архивное отображение | точные поля и переходы не полностью раскрыты |
| Документ | добавление → просмотр/скачивание/печать → редактирование → удаление | нет версий, ЭП, согласования, журналов доступа |
| Комплект документов | заполнение мастера → проверка шагов → формирование → скачивание пакета/отдельных документов → повторное формирование после изменений | нет версионирования пакетов кроме дат изменения/формирования |
| Объект | добавить объект → карточка объекта → технические устройства/специалисты/оборудование → заключения | недостаточно данных |
| Заключение | добавить заключение → список → просмотр копии → скачивание → удаление | недостаточно данных по полям, статусам, подписанию, нумерации, архиву |

## 15. Файлы, вложения и архив

| Аспект | Вывод по PDF |
|---|---|
| Где используются файлы | документы организации и лаборатории, паспорт/сертификат оборудования, копии поверок/калибровок, копии удостоверений НК/ПБ, допуски/разрешения, фото и подпись специалиста, комплект документов, копии заключений. |
| Поддерживаемые форматы | Для паспорта/сертификата оборудования на скриншоте видны PDF, JPG, PNG, TIFF до 20 МБ. Для фото/подписи специалиста видны JPG или PNG до 10 МБ. Для остальных документов PDF говорит о предпросмотре поддерживаемых форматов, но полный список не указан. |
| Просмотр | Если формат поддерживает предпросмотр, файл открывается в новом окне. Если нет — доступен для скачивания и локального просмотра. |
| Скачивание | Документы можно скачать; комплекты и списки с копиями скачиваются архивом zip. |
| Печать | Для документов с доступным просмотром есть печать. |
| Версии файлов | Не описаны. При замене или удалении файла старый файл удаляется безвозвратно. |
| Архив | Отдельный электронный архив как модуль не описан. Есть архив поверок и хранение документов. В разделе документов упомянуты инструкции по ведению архива как формируемые документы. |
| Электронная подпись | Не описана. Есть файл подписи специалиста на карточке, но это не равнозначно юридически значимой электронной подписи. |
| Журнал действий с файлами | Не описан. |
| Ограничения хранения | Для отдельных типов файлов ограничения размера видны на скриншотах; общая политика хранения не описана. |

Недостаточно данных для точного вывода о физическом хранении файлов: файловая система, объектное хранилище, БД, шифрование, антивирусная проверка, дедупликация, версии, резервное копирование файлов отдельно от БД в PDF не раскрыты.

## 21. Риски использования документации конкурента

| Риск | Суть риска | Как учитывать при проектировании |

| Некорректный перенос чужой логики | Логика Lab365 может быть заточена под их продукт, клиентов, шаблоны и справочники. | Использовать только как аналитический ориентир; подтверждать процессы у собственного заказчика. |

| Неверная интерпретация скриншотов | Скриншоты могут быть устаревшими или демонстрационными. | Все поля со скриншотов помечать как источник низкой уверенности, если они не описаны текстом. |
| Смешение документации с требованиями | Руководство пользователя не является ТЗ будущей системы. | Каждое требование помечать статусом: явно, следует, предположение. |
| Устаревание документации | PDF обновлен 30.11.2023; продукт мог измениться. | Не считать документ актуальным описанием текущей версии без проверки. |
| Юридически нежелательное копирование | Нельзя копировать формулировки, структуру экранов, дизайн, шаблоны документов. | Проектировать собственные процессы, названия и интерфейсы; не воспроизводить материалы дословно. |
| Нормативная зависимость | Указанные в PDF нормативные ссылки могут не совпадать с регламентами заказчика. | Уточнить применимые стандарты, отрасль, требования аттестации и документы. |
| Недооценка эксплуатации | В PDF мало технических деталей о хранении файлов, бэкапах, интеграциях, аудитах. | Проработать эксплуатационную архитектуру отдельно. |

## 23. Что можно использовать как ориентир при проектировании

1. Модульное разделение: организация, лаборатория, пользователи, склад, специалисты, документы, объекты, заключения.
2. Идею системообразующих данных: документы и заключения должны формироваться из нормализованных карточек организации, лаборатории, специалистов и оборудования.
3. Ролевую модель как стартовую основу, но не как окончательную ACL.
4. Учет сроков действия поверок/калибровок и удостоверений с уведомлениями.
5. Блокировку использования оборудования без действующей поверки/калибровки при оформлении заключений.
6. Блокировку участия специалиста без актуальной области аттестации.
7. Отказ от физического удаления оборудования и специалистов в пользу статусов «списано» и «уволен».
8. Мастер формирования сложного комплекта документов с пошаговой проверкой достаточности данных.
9. Разделение загружаемых документов и документов, формируемых системой по запросу.
10. Генерацию DOCX и ZIP-архивов с приложенными копиями документов.
11. Индивидуальную прочитанность уведомлений для каждого пользователя.
12. Использование иерархии методов/подметодов и объектов/подобъектов контроля.

## 24. Что нельзя считать подтвержденным требованием

1. Нельзя считать подтвержденной структуру БД Lab365.
2. Нельзя считать подтвержденными API-методы, форматы обмена и интеграционные события.
3. Нельзя считать подтвержденным полный жизненный цикл объекта испытаний и заключения.
4. Нельзя считать подтвержденными статусы заявок, заданий, работ, результатов и заключений.
5. Нельзя считать подтвержденным наличие аудита действий.
6. Нельзя считать подтвержденным наличие электронной подписи.
7. Нельзя считать подтвержденным версионирование документов и файлов.
8. Нельзя считать подтвержденной модель подрядных лабораторий и корпоративной иерархии доступа.
9. Нельзя считать подтвержденной структуру шаблонов документов.
10. Нельзя считать подтвержденным, что все функции Lab365 нужны в собственной системе.
11. Нельзя считать подтвержденным, что скриншоты соответствуют актуальной версии продукта.
12. Нельзя считать подтвержденными поля модулей «Объекты и заключения», потому что подробное описание отсутствует в предоставленном PDF.

## 25. Выводы для дальнейшей разработки

PDF дает хорошую основу для понимания того, какие блоки нужны зрелой системе для испытательной лаборатории: пользователи и роли, организация, лаборатория, оборудование, поверки, специалисты, удостоверения, документы, уведомления, справочники, аттестационный комплект, объекты и заключения.

Для собственной системы наиболее полезны три вывода.

Первый: оперативный модуль заключений нельзя проектировать изолированно. Он должен опираться на нормализованные данные специалистов, оборудования, документов, поверок, удостоверений и области контроля.

Второй: контроль сроков действия поверок, калибровок и удостоверений — не второстепенная функция. В PDF эта логика напрямую влияет на возможность использовать оборудование и специалистов в работах и документах.

Третий: предоставленный PDF почти не раскрывает самый важный для рабочей NDT-системы модуль «Объекты и заключения». Поэтому для проектирования БД, API и интерфейсов по заявкам, объектам, техническим устройствам, элементам контроля, заключениям и журналам обязательно нужны дополнительные источники: реальные документы заказчика, Excel-учет, формы заключений, примеры журналов и описание фактического процесса.
