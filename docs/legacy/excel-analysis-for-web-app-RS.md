# Анализ Excel-учета для проектирования веб-приложения

## 1. Назначение документа

Документ фиксирует фактическую структуру и данные Excel-файла `RS.xlsx`, который используется как рабочий учет НК/NDT. Цель — восстановить по таблице текущий учетный процесс и подготовить основу для проектирования веб-приложения: доменной модели, базы данных, API, интерфейсов, импорта исторических данных и архивного хранения.

В документе разделяются факты из Excel, логические выводы и предположения. Там, где данных недостаточно, это указано явно.

## 2. Исходные материалы

| Материал | Роль в анализе | Статус использования |
| --- | --- | --- |
| RS.xlsx | Основной источник фактов: структура книги, листы, колонки, формулы, значения, стили, фильтры, ошибки и связи. | Использован как основной источник. |
| deep-research-report.md | Дополнительный предметный контекст по НК сварных швов, ролям, документам, жизненному циклу заявок и архиву. | Использован только как вспомогательный контекст. Выводы по конкретной системе сделаны по Excel. |

Ограничение: в Excel нет явных описаний бизнес-правил, инструкций заполнения, утвержденных форм документов, матрицы ролей и справочников. Поэтому часть бизнес-процесса восстановлена как логический вывод или предположение.

## 3. Краткое резюме анализа

- Файл `RS.xlsx` содержит один видимый лист `RS`. Скрытых листов в книге не обнаружено.
- Лист `RS` — это плоский рабочий журнал/реестр, объединяющий заявки, стыки, чертежи, линии, параметры сварки, методы НК, результаты/статусы РК, данные по пленкам, передачам, архиву, а также отдельные блоки по ВИК, ПВК и МК.
- В листе 4724 строк и 88 колонок. Первая строка — заголовки. Строк данных — 4723.
- Автофильтр установлен на диапазон `A1:CJ4681`, но в книге есть 43 заполненные строки после конца фильтра: [4682, 4683, 4684, 4685, 4686]...[4720, 4721, 4722, 4723, 4724]. Это риск: такие строки могут не участвовать в фильтрации и ручной работе.
- Найдено 42639 формул. Часть формул генерирует составные ключи и расчетные значения, часть содержит ошибки `#REF!`, `#N/A` или зависит от внешней книги `Расчет времени эксп. D4-D7.xlsx`.
- Основной рабочий результат в колонке `AA / Результат` содержит не итоговую годность стыка, а статусы пересвета: `ПЕРЕСВЕТ`, `ПЕРЕСВЕТ ПЕРЕДЕЛКИ`, `ПЕРЕСВЕТ НОЧЬ`. Следовательно, таблица отражает не только итоговый контроль, но и операционный процесс радиографического пересвета/повторной съемки.
- Блоки `BF:BL`, `BM:BS`, `BT:BZ` отражают ВИК, ПВК и МК: номера заключений, даты, результаты, дефекты, реестры передачи и акты ВР. Эти блоки выглядят как присоединенные к основному РК-реестру данные по другим методам контроля.
- В таблице нет явных заказчиков, договоров, пользователей системы, ролей, оборудования, файлов, подписей, схем и утвержденных маршрутов согласования. Эти сущности нужны для веб-приложения, но должны проектироваться как выводы из структуры Excel и уточняться у заказчика.

## 10. Связи между сущностями

- `Заявка НК` 1:N `Сварной стык` — подтверждается повторением одного `Номер заявки` на множестве строк.
- `Чертеж` 1:N `Сварной стык` — один номер чертежа встречается в разных строках с разными стыками.
- `Линия` 1:N `Сварной стык` — номер линии и номер стыка вместе образуют объект контроля.
- `Сварной стык` N:M `Метод контроля` — колонка `V` содержит комбинации методов: `ВИК,РК,МК`, `ВИК,РК,ПВК`, `РК`, `ВИК,РК` и др.
- `Метод контроля` 1:N `Заключение` — по каждому методу есть свои номера/даты заключений в отдельных блоках.
- `РК` 1:N `Снимок/пленка` — колонка `AC / Кол-во снимков` и расчет `CC/CD` показывают количество и расчет пленки.
- `Заключение` N:1 `Реестр передачи` — номера реестров группируют множество строк.
- `Заключение/пленка` N:1 `Архивный реестр` — колонки `AI`, `BC` указывают передачу в архив.
- `Сварной стык` N:2 `Сварщик` — по строке есть два клейма сварщиков Q/R.
- `Результат контроля` N:1 `Дефектоскопист` — по РК есть `AD / Клеймо деф-та`. Для ВИК/ПВК/МК явного исполнителя в отдельных блоках нет.

## 11. Восстановленный бизнес-процесс

### Факты из Excel

1. Запись начинается с заявки: есть `Дата заявки` и `Номер заявки`.
2. В строке указываются объектные данные: титул, чертеж, лист, ревизия, линия, стык, диаметр, толщина, материалы.
3. Указываются сварочные данные: дата сварки, PWHT, клейма сварщиков, способ сварки, тип шва.
4. Указываются требуемые методы НК: ВИК, РК, ПВК, МК в разных комбинациях.
5. Для РК фиксируются номер заключения, дата контроля, рабочий статус пересвета, коды/описание дефектов, количество снимков, клеймо дефектоскописта, местоположение, дата заключения/баркода.
6. Далее фиксируются реестры передачи пленок/заключений, архив, акт ВР.
7. Дополнительно фиксируются ВИК/ПВК/МК: заключение, дата, результат, дефекты, реестр верификации ТН, передача в ПТО, акт ВР.

### Логический вывод по процессу

Процесс в Excel выглядит так:

```text
Заявка → перечень стыков по чертежам/линиям → сварочные данные → назначенные методы НК → РК/ВИК/ПВК/МК → заключения → реестры передачи → ПТО/SEG/архив → акты ВР
```

Особенность файла: основной операционный фокус — радиографический контроль и пересветы. Это видно по `AA / Результат`, где все значения являются вариантами `ПЕРЕСВЕТ`, а также по количеству снимков, пленке, плотностям, досветам и реестрам передачи пленок.

### Предположения

- Заявку создает внешний заказчик или ответственное подразделение, а в Excel ее регистрирует оператор/специалист НК. В Excel нет поля автора заявки.
- Распределение дефектоскопистам отражено не отдельным заданием, а через `Клеймо деф-та` и даты контроля/заключения.
- Результаты ВИК/ПВК/МК, вероятно, подтягивались из отдельных журналов или оформлялись параллельно, потому что в заголовке `BF` видна ссылка на формулу ВПР по листу `ВИК`, но самого листа в книге нет.
- Передачи `SEG`, `ТН`, `ПТО` и `архив` являются этапами документооборота, а не технического контроля как такового.

| ВИК | не требуется/выполнен → годен → передан в ПТО → акт ВР | BH, BK, BL | Частично факт |
| ПВК | не требуется/выполнен → годен → передан в ПТО → акт ВР | BO, BR, BS | Частично факт |
| МК | не требуется/выполнен → годен → передан в ПТО → акт ВР | BV, BY, BZ | Частично факт |
| Заключение | создано → датировано → передано → архивировано | номера/даты заключений, реестры передачи | Предположение по документообороту |

## 13. Роли пользователей и матрица ответственности

В Excel явно видны только клейма сварщиков и дефектоскопистов. Персональные роли пользователей системы, ФИО операторов, начальников, лаборантов, подписантов и заказчиков в файле отсутствуют.

| Роль | Подтверждение в Excel | Данные/действия | Статус |
| --- | --- | --- | --- |
| Оператор/регистратор заявки | Номер и дата заявки есть, но автор не указан. | Ввод заявки, чертежей, стыков, импорт Excel. | Предположение |
| Начальник участка / руководитель работ | Прямого поля нет. Логически нужен для распределения работ и контроля статусов. | Назначение методов/исполнителей, контроль выполнения. | Предположение |
| Дефектоскопист РК | `AD / Клеймо деф-та` содержит клейма исполнителей. | Съемка/пересвет, ввод количества снимков, данных пленки, статусов. | Факт наличия клейма, права — предположение |
| Сварщик | `Q/R` клейма сварщиков. | Не пользователь НК-системы обязательно; связан со стыком. | Факт как объект учета |
| Специалист НК / инженер НК | Прямого поля нет; нужен для интерпретации результатов и заключений. | Проверка результатов, оформление заключений, исправления статусов. | Предположение |
| Лаборант/архивариус | Прямого поля нет; архив и передачи видны через реестры. | Прием материалов, реестры, архив. | Предположение |
| ПТО / ТН / SEG | Названия в заголовках реестров передачи. | Получение/верификация/передача заключений. | Факт как участники документооборота, роли требуют уточнения |
| Администратор системы | В Excel нет. | Пользователи, права, справочники, аудит, импорт. | Техническая роль |

Аудит обязателен для изменений: номера заключений, даты контроля/заключения, результаты, дефекты, реестры, акты, архивные номера, исправления статусов и все поля, влияющие на документы.

## 15. Файлы, вложения и архив

В самом Excel вложений, гиперссылок, файлов или изображений не обнаружено. Но структура полей показывает необходимость файлового архива:

- радиографические снимки/пленки, связанные с `Баркод`, `Кол-во снимков`, `Пленка`, `Итог`;
- сканы или PDF заключений РК/ВИК/ПВК/МК;
- реестры передачи SEG/ТН/ПТО/архив;
- акты ВР;
- исходный Excel-файл импорта и лог сопоставления строк;
- чертежи/изометрии, на которые ссылаются `№ чертежа`, `Лист`, `Rev`;
- возможные дефектограммы/фото — в Excel не представлены, поэтому это предположение для будущей системы.

Для импорта важно хранить оригинальный `RS.xlsx`, номер строки Excel, исходные значения всех 88 колонок и результат нормализации.

## 16. Требования к веб-приложению

### Функциональные требования

- Ведение заявок НК с датой, номером, списком стыков и требуемыми методами контроля.
- Учет сварных стыков: чертеж, лист, ревизия, линия, номер стыка, диаметр, толщина, материалы, дата сварки, PWHT, сварщики, способ сварки, тип шва, категория трубопровода.
- Учет методов контроля по стыку: РК, ВИК, ПВК, МК. Метод должен быть отдельной сущностью, а не строкой через запятую.
- Учет РК: заключение, дата контроля, статус пересвета, дефекты/коды, количество снимков, дефектоскопист, местоположение, дата заключения/баркода, пленка, итог, плотности/досветы.
- Учет ВИК/ПВК/МК: номер заключения, дата, результат, дефекты, реестры, акты.
- Реестры передачи: SEG, ТН, ПТО, архив.
- Поиск и фильтрация по заявке, титулу, чертежу, линии, стыку, баркоду, заключению, методу, статусу, дефектоскописту, датам, реестрам и актам.
- Импорт исторического Excel с сохранением исходной строки.
- Журнал исправлений и аудит изменений для всех значимых полей.

### Требования к данным
- Нормализовать справочники: методы контроля, материалы, способы сварки, типы швов, PWHT, НТД, клейма сварщиков, клейма дефектоскопистов, статусы, реестры/типы документов.
- Разделить рабочие данные и расчетные поля. Например, `CC/CD` должны вычисляться, а не редактироваться вручную.
- Сохранять исходные значения из Excel для трассировки.
- Поддержать неоднозначность: один стык может иметь несколько строк из-за пересветов/переделок/повторных документов.

### Требования к импорту Excel
- Импорт должен иметь предварительную валидацию: строки за пределами фильтра, пустые баркоды, пустые заключения, ошибки формул, смешанные даты, дубли.
- Для каждой импортированной строки хранить `import_batch_id`, номер листа, номер строки, хеш исходных значений, JSON исходной строки и результат нормализации.
- Ошибочные формулы не пересчитывать слепо: `#REF!`, внешние ссылки и сломанные ВПР должны попадать в отчет импорта.

### Требования к безопасности, эксплуатации и архиву
- Ролевой доступ к редактированию результатов, заключений, реестров и архивных записей.
- Запрет бесследного изменения результатов и номеров документов.
- Версионирование документов и фиксирование исправлений.
- Резервное копирование БД и файлового архива.
- Возможность выгрузить карточку стыка/заявки со всеми связанными документами и исходной строкой Excel.

## 18. Импорт исторических данных из Excel

### Что можно импортировать напрямую
- Заявки: `Дата заявки`, `Номер заявки`.
- Объектные данные: титул, чертеж, лист, ревизия, линия, стык, диаметр, толщина.
- Материалы, клейма сварщиков, способ сварки, тип шва, категория трубопровода — после справочников и нормализации.
- Номера и даты заключений, реестры, акты — после разбиения по методам.
- Количество снимков и расчетные данные пленки — после проверки формул.

### Что нужно очистить до импорта
- Пробелы и регистр: `ПЕРЕСВЕТ`, ` ПЕРЕСВЕТ`, `ПЕРЕСВЕТ `; `н/п`, `Н/П`, `Н/П `.
- Кириллица/латиница: `РК`, `PК`, `RK`; `МК`, `MK`; `C17`, `С17`.
- Ошибки формул: `#REF!`, `#N/A`, `#VALUE!`.
- Даты, сохраненные как числа/время `00:00:00`.
- Колонки с противоречивым смыслом: `AQ`, `AR`, `BI`, `BJ`, `BA`, `BD`, `BE`.
- Строки 4682–4724 за пределами автофильтра.

### Рекомендуемый алгоритм импорта

1. Сохранить оригинальный `RS.xlsx` как файл импорта.
2. Создать запись `import_batches`.
3. Прочитать лист `RS`, каждую строку сохранить в `import_rows.raw_json` без потери исходных значений.
4. Сформировать отчет по ошибкам: пустые баркоды, пустые заключения, ошибки формул, даты, дубли, строки вне автофильтра.
5. Создать справочники: материалы, методы, статусы, способы сварки, типы швов, PWHT, НТД, клейма.
6. Импортировать заявки.
7. Импортировать чертежи, линии и стыки.
8. Разобрать `V / Вид контроля` на отдельные методы.
9. Импортировать результаты РК, затем ВИК/ПВК/МК.
10. Импортировать заключения, реестры и акты.
11. Связать все сущности с исходной строкой Excel.
12. Подготовить отчет расхождений для ручной проверки.

Откат импорта должен удалять или помечать все сущности, созданные конкретным `import_batch_id`, без потери оригинального файла и протокола ошибок.

## 20. Риски проектирования и миграции

- Риск неверной интерпретации `AA / Результат` как итоговой годности. По факту это статус пересвета.
- Риск потери истории пересветов при схлопывании строк в один стык.
- Риск потери связей между заявкой, чертежом, линией и стыком при выборе неправильного уникального ключа.
- Риск импорта мусорных расчетных значений из сломанных формул.
- Риск неправильной нормализации методов контроля из-за латиницы/кириллицы и пробелов.
- Риск неверной работы фильтров из-за строк 4682–4724 за пределами автофильтра.
- Риск проектирования лишних ролей без подтверждения у заказчика.
- Риск отсутствия архива файлов: Excel содержит ссылки на пленки/заключения/реестры по номерам, но не содержит сами файлы.
- Риск смешения справочников и рабочих документов: номера заключений и реестров не должны быть справочниками.
- Риск проектирования БД без учета повторного контроля, пересвета и переделок.

## 22. Выводы для дальнейшей разработки

1. Будущая система должна проектироваться не как импорт одной Excel-таблицы, а как нормализованная система учета заявок, стыков, методов контроля, результатов, заключений, реестров, актов и архива.
2. Основная сущность для трассировки — не только `заявка`, но связка `заявка → чертеж → линия → стык → метод контроля → результат/заключение → реестр/архив`.
3. РК должен иметь отдельную метод-специфичную модель: снимки, баркоды, пленка, пересветы, плотности, досветы, дефектоскопист, архив.
4. ВИК, ПВК и МК должны быть отдельными результатами контроля по тому же стыку, а не колонками одной строки.
5. Импорт нужно делать через слой `raw import rows`, иначе невозможно безопасно откатить, проверить и объяснить перенос исторических данных.
