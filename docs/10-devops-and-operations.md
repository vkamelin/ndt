# Эксплуатация и DevOps

## 1. Назначение

Документ фиксирует production-инфраструктуру, очереди, scheduler, backup, логи и эксплуатационные правила.

## 2. Базовая инфраструктура

Production-стек:

- Ubuntu Server LTS;
- Nginx;
- PHP-FPM;
- MySQL 8.4;
- Redis;
- Supervisor;
- cron;
- S3-compatible хранилище;
- Laravel logs;
- системные логи.

Docker Compose используется для разработки.

## 3. Что не использовать как базовое решение

Не закладывать без прямой необходимости:

- Kubernetes;
- микросервисы;
- Octane;
- Swoole;
- сложную multi-node инфраструктуру;
- отдельные сервисы для каждой доменной области.

## 4. Очереди

Через Laravel Queue должны выполняться:

- тяжелые Excel-экспорты;
- генерация PDF;
- отправка уведомлений;
- обработка больших файлов;
- массовые операции;
- операции, которые могут выполняться дольше обычного HTTP-запроса.

Supervisor должен контролировать queue worker.

Пример процессов:

```text
laravel-worker-default
laravel-worker-reports
laravel-worker-notifications
```

Разделение очередей допустимо, если оно реально нужно.

## 5. Scheduler

Laravel Scheduler должен запускаться через cron.

Планируемые задачи:

- проверка просроченных заданий;
- уведомления о незавершенных сменах;
- уведомления о сроках поверки/калибровки;
- уведомления о сроках удостоверений;
- очистка временных файлов;
- служебные проверки очередей;
- контроль backup, если это реализовано внутри приложения.

## 6. Backup

Production должен иметь:

- ежедневный backup MySQL;
- ежедневный backup файлов;
- хранение нескольких последних копий;
- проверку успешности backup;
- отдельный лог ошибок backup;
- инструкцию восстановления.

Backup БД и файлов должен быть согласован по времени настолько, насколько это возможно в простой инфраструктуре.

## 7. Логи

Обязательные источники логов:

- Laravel application logs;
- queue worker logs;
- scheduler logs;
- Nginx access/error logs;
- PHP-FPM logs;
- MySQL logs;
- backup logs;
- audit log в БД.

Нельзя заменять audit log обычным текстовым логом.

Audit log — часть предметной системы.

## 8. Health-check

Нужен health-check endpoint.

Минимально он должен проверять:

- приложение отвечает;
- БД доступна;
- Redis доступен;
- storage доступен.

Не раскрывать чувствительные технические данные в публичном health-check.

## 9. Мониторинг

Минимальный мониторинг:

- место на диске;
- состояние очередей;
- ошибки Laravel;
- ошибки queue worker;
- успешность backup;
- доступность БД;
- доступность Redis;
- размер storage.

## 10. Файловое хранилище

Файлы хранятся в приватном storage или S3-compatible хранилище.

Правила:

- не хранить приватные файлы в публичной директории;
- не отдавать прямые пути;
- скачивание только через backend с проверкой прав;
- для временного доступа использовать signed URL.

## 11. Обновления

Порядок обновления production:

1. Сделать backup БД и файлов или проверить наличие свежего backup.
2. Получить код.
3. Установить зависимости.
4. Выполнить миграции.
5. Собрать frontend assets.
6. Очистить/пересобрать кэши Laravel.
7. Перезапустить PHP-FPM при необходимости.
8. Перезапустить queue workers.
9. Проверить health-check.
10. Проверить основные сценарии.

## 12. Миграции

Правила:

- миграции должны быть обратимыми, если это возможно;
- миграции не должны терять данные без прямого решения;
- большие миграции данных не выполнять внутри обычного HTTP-запроса;
- индексы добавлять осознанно под реальные фильтры;
- FK добавлять для обязательных связей.

## 13. .env

В `.env` должны быть настройки:

- приложения;
- БД;
- Redis;
- queue;
- mail;
- filesystem disk;
- S3-compatible хранилища;
- trusted proxies, если применимо.

Не коммитить реальные секреты.

## 14. Production checklist

Перед production должны быть проверены:

- HTTPS;
- `APP_ENV=production`;
- `APP_DEBUG=false`;
- миграции выполнены;
- seed базовых ролей и справочников выполнен;
- queue worker работает;
- scheduler работает;
- storage доступен;
- backup настроен;
- health-check отвечает;
- права на storage корректны;
- приватные файлы не доступны напрямую;
- базовые роли работают;
- администратор видит все;
- начальник участка ограничен своим объектом/участком.
