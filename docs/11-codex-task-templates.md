# Шаблоны задач для Codex

## 1. Назначение

Документ содержит готовые формулировки задач для Codex. Использовать их как основу, когда нужно поручить агенту реализацию отдельной части проекта.

## 2. Общий шаблон задачи

```text
Изучи README.md, AGENTS.md, project.md и профильные документы из docs/.

Задача: <кратко описать задачу>.

Ограничения:
- не менять утвержденный стек;
- не добавлять новые сущности, поля, роли и статусы без необходимости;
- не менять соседние модули без причины;
- бизнес-логику держать в сервисах;
- контроллеры оставить тонкими;
- доступ проверять через policies;
- критичные действия писать в audit log;
- добавить или обновить тесты, если меняется бизнес-логика.

Результат:
- внести изменения в код;
- добавить/обновить миграции, модели, сервисы, FormRequest, policies, Livewire/Blade при необходимости;
- добавить/обновить тесты;
- кратко описать, что изменено.
```

## 3. Задача на создание модуля

```text
Изучи README.md, AGENTS.md, project.md и docs/01-architecture.md.

Создай модуль <ModuleName> для <назначение модуля>.

Модуль должен соответствовать архитектуре модульного монолита Laravel.

Минимально подготовь:
- модели;
- миграции;
- enum-классы для статусов/типов, если нужны;
- FormRequest;
- сервисы;
- policies;
- маршруты;
- базовые Blade/Livewire-экраны, если задача включает UI;
- feature/unit тесты для критичных сценариев.

Не добавляй лишние функции сверх задачи.
```

## 4. Задача на миграции и модели

```text
Изучи docs/03-data-model.md и project.md.

Создай миграции и модели для <перечень сущностей>.

Требования:
- использовать FK для обязательных связей;
- добавить индексы под фильтры, указанные в документации;
- не хранить файлы в БД;
- не хранить методы контроля строкой;
- использовать soft delete или статус аннулирования для рабочих сущностей, если данные могут участвовать в документах;
- добавить factories, если они нужны для тестов.

После реализации проверь миграции и тесты.
```

## 5. Задача на роли и доступ

```text
Изучи docs/04-roles-and-access.md, AGENTS.md и project.md.

Реализуй роли, permissions и policies для <сущность/модуль>.

Обязательные правила:
- администратор видит все;
- начальник участка работает только со своим объектом/участком;
- исполнитель видит только назначенные ему задания/материалы, если иное не задано;
- backend проверяет доступ через policy;
- UI не заменяет backend-проверку;
- изменение ролей и объекта/участка сотрудника писать в audit log.

Добавь тесты на разрешенный и запрещенный доступ.
```

## 6. Задача на workflow

```text
Изучи docs/05-business-workflows.md и project.md.

Реализуй workflow <название workflow>.

Требования:
- статусы оформить enum-классами;
- переходы статусов выполнять через сервисы;
- запретить недопустимые переходы;
- критичные переходы писать в audit log;
- добавить feature-тесты успешных и запрещенных сценариев;
- не размещать бизнес-логику в контроллере или Livewire-компоненте.
```

## 7. Задача на UI

```text
Изучи docs/06-ui-map.md, AGENTS.md и project.md.

Реализуй UI для <раздел/экран>.

Требования:
- Blade + Livewire 3 + Alpine.js + Tailwind CSS;
- серверная пагинация для списков;
- фильтры на стороне БД;
- действия показывать с учетом прав;
- backend-проверки доступа обязательны;
- UI должен быть простым, быстрым и рабочим;
- не добавлять тяжелую SPA-архитектуру.
```

## 8. Задача на API

```text
Изучи docs/07-api-guidelines.md, AGENTS.md и project.md.

Реализуй API для <PWA/mobile-сценарий>.

Требования:
- использовать Sanctum;
- использовать rate limiting;
- проверять права через policies;
- использовать FormRequest/DTO;
- возвращать единый формат ошибок;
- не отдавать приватные storage-пути;
- критичные действия писать в audit log;
- добавить feature-тесты.
```

## 9. Задача на файлы

```text
Изучи docs/08-files-documents-audit.md и project.md.

Реализуй работу с файлами для <сущность>.

Требования:
- файлы хранить приватно;
- в БД хранить только метаданные;
- скачивание только через backend с проверкой прав;
- не отдавать прямой storage path;
- удаление/аннулирование файла писать в audit log;
- добавить тесты загрузки, скачивания и запрета скачивания без прав.
```

## 10. Задача на заключения

```text
Изучи docs/05-business-workflows.md, docs/08-files-documents-audit.md и project.md.

Реализуй <создание/утверждение/версионирование/аннулирование> заключений.

Требования:
- утвержденное заключение нельзя редактировать напрямую;
- исправления только через новую версию, замену, аннулирование или корректирующую запись;
- PDF-генерацию выполнять через очередь, если операция тяжелая;
- критичные действия писать в audit log;
- добавить тесты на запрет прямого редактирования утвержденного заключения.
```

## 11. Задача на смену лаборанта

```text
Изучи docs/05-business-workflows.md и project.md.

Реализуй смену лаборанта.

Сценарии:
- старт смены;
- прием пленки;
- выдача пленки;
- списание пленки;
- поступление химии;
- замена химии;
- запрос химии;
- регламентные работы;
- передача материалов на дешифровку;
- завершение смены.

Требования:
- один сотрудник не должен иметь две открытые смены лаборанта одновременно;
- завершение смены требует обязательные данные;
- критичные действия писать в audit log;
- добавить feature-тесты.
```

## 12. Задача на смену дешифровщика

```text
Изучи docs/05-business-workflows.md и project.md.

Реализуй смену дешифровщика.

Сценарии:
- старт смены;
- получение списка пленок/снимков;
- фиксация просмотренных групп;
- фиксация брака;
- фиксация подозрения на подлог;
- фиксация очистки рабочего места;
- внесение результатов дешифровки;
- передача результатов на анализ;
- завершение смены.

Требования:
- один сотрудник не должен иметь две открытые смены дешифровщика одновременно;
- завершение смены требует журнал просмотренной пленки и отметку об очистке рабочего места;
- критичные действия писать в audit log;
- добавить feature-тесты.
```

## 13. Задача на тесты

```text
Изучи docs/09-testing-strategy.md.

Добавь тесты для <модуль/сценарий>.

Проверь:
- успешный сценарий;
- запрещенный сценарий;
- права доступа;
- изменение статусов;
- audit log, если действие критичное;
- работу с файлами, если они участвуют в сценарии.
```
