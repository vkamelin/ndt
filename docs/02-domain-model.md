# Предметная модель

## 1. Назначение

Документ фиксирует ключевые предметные сущности проекта и связи между ними. Он нужен Codex, чтобы не восстанавливать доменную модель заново при каждой задаче.

## 2. Базовая иерархия

```text
Город
→ Объект / участок
→ Заявка
→ Стык
→ Метод контроля
→ Задание
→ Результат
→ Заключение
→ Реестр / архив
```

Каждый сотрудник закреплен за конкретным объектом/участком.

Объект/участок относится к городу.

В одном городе может быть один или несколько объектов/участков.

## 3. Город

Город — территориальная группировка объектов/участков.

Минимальные данные:

- название;
- активность;
- комментарий.

Город нужен для фильтрации, отчетов и группировки объектов/участков.

## 4. Объект / участок

Объект/участок — рабочая производственная единица в конкретном городе.

В проекте участок и объект используются как близкие понятия: участок в системе — это объект в определенном городе.

Минимальные данные:

- город;
- название объекта/участка;
- код или короткое обозначение, если будет утверждено;
- активность;
- комментарий.

Связи:

- объект/участок относится к городу;
- сотрудник закрепляется за объектом/участком;
- заявка относится к объекту/участку;
- стык относится к объекту/участку;
- оборудование относится к объекту/участку;
- смена открывается на объекте/участке.

## 5. Сотрудник

Сотрудник — производственная сущность. Пользователь — учетная запись для входа.

Сотрудник должен хранить:

- ФИО;
- должность;
- город;
- объект/участок закрепления;
- телефон;
- email;
- статус;
- табельный номер;
- связанные документы.

Связи:

- сотрудник может быть связан с пользователем;
- сотрудник имеет квалификации;
- сотрудник может быть исполнителем задания;
- сотрудник может открывать смену;
- сотрудник может быть указан в результатах контроля и заключениях.

## 6. Пользователь

Пользователь нужен для входа и доступа к системе.

Данные пользователя:

- имя;
- email;
- пароль;
- статус;
- роли;
- связь с сотрудником.

Статусы:

- active;
- blocked.

Пользователь со статусом `blocked` не должен работать в системе.

## 7. Заявка НК

Заявка фиксирует входящий объем работ.

Данные заявки:

- номер заявки;
- дата заявки;
- заказчик;
- город;
- объект/участок;
- титул;
- ответственное подразделение;
- приоритет;
- срок выполнения;
- основание работ;
- комментарий;
- файлы;
- список стыков;
- требуемые методы контроля.

Статусы заявки:

- черновик;
- зарегистрирована;
- на уточнении;
- принята в работу;
- запланирована;
- в работе;
- частично выполнена;
- ожидает анализа;
- на утверждении;
- выдана;
- закрыта;
- отменена.

## 8. Стык

Стык — центральная производственная сущность.

Данные стыка:

- город;
- объект/участок;
- титул;
- чертеж;
- лист;
- ревизия;
- линия;
- номер стыка;
- диаметр;
- толщина;
- материал 1;
- материал 2;
- дата сварки;
- сварщики и их клейма;
- способ сварки;
- тип шва;
- категория трубопровода;
- среда;
- PWHT;
- НТД / норма оценки;
- требуемые методы контроля;
- текущий статус.

Статусы стыка:

- создан;
- ожидает контроля;
- назначен в задание;
- в контроле;
- контроль выполнен;
- ожидает анализа;
- годен;
- дефект;
- требуется ремонт;
- в ремонте;
- ожидает повторного контроля;
- исключен;
- закрыт.

## 9. Метод контроля

Метод контроля должен быть отдельной сущностью или связью, а не строковым списком внутри стыка.

Базовые методы:

- РК — радиографический контроль;
- ВИК — визуальный и измерительный контроль;
- ПВК — капиллярный контроль;
- МК — магнитопорошковый контроль;
- УК — ультразвуковой контроль.

Один стык может иметь несколько методов контроля.

## 10. Задание

Задание связывает заявку, стыки, метод контроля, исполнителя, срок и статус выполнения.

Данные задания:

- номер задания;
- заявка;
- город;
- объект/участок;
- метод контроля;
- исполнитель;
- список стыков;
- плановая дата;
- приоритет;
- комментарий начальника участка;
- статус.

Статусы задания:

- создано;
- назначено;
- принято исполнителем;
- в работе;
- выполнено;
- выполнено частично;
- возвращено;
- отменено.

## 11. Результат контроля

Результат фиксирует итог выполнения метода контроля по стыку.

Общие данные результата:

- стык;
- метод контроля;
- исполнитель;
- дата контроля;
- оборудование;
- НТД;
- результат;
- дефекты;
- файлы;
- связь с заключением;
- статус;
- история изменений.

Для РК используется расширенная модель: пленки, снимки, пересветы, плотности, дешифровка, архив.

## 12. Заключение

Заключение — официальный итоговый документ по одному или нескольким результатам контроля.

Данные заключения:

- номер;
- дата;
- метод контроля;
- город;
- объект/участок;
- заявка;
- список стыков;
- НТД;
- исполнитель;
- проверяющий;
- утверждающий;
- результат;
- дефекты;
- приложенные файлы;
- статус.

Статусы заключения:

- черновик;
- подготовлено;
- на проверке;
- возвращено на доработку;
- утверждено;
- выдано;
- аннулировано;
- заменено.

Утвержденное заключение нельзя редактировать напрямую.

## 13. Смена

В системе есть два типа смен:

- смена лаборанта;
- смена дешифровщика.

Смена создается для конкретного сотрудника, даты, объекта/участка и типа смены.

Один сотрудник не должен иметь две открытые смены одного типа одновременно.

## 14. Реестр и архив

Реестр фиксирует передачу пленок, заключений, документов или архивных материалов.

Архив должен хранить связь между:

- заявкой;
- стыком;
- методом контроля;
- заключением;
- пленкой/снимком;
- реестром;
- файлами.
