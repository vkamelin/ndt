# Файлы, документы, версии и audit log

## 1. Назначение

Документ фиксирует правила работы с файлами, документами, версиями и журналом действий.

## 2. Общие правила файлов

1. Файлы хранятся приватно.
2. Скачивание файлов выполняется только через backend.
3. Перед скачиванием backend проверяет права доступа к связанной сущности.
4. Прямые storage-пути нельзя отдавать пользователю.
5. Файлы нельзя хранить в БД.
6. В БД хранятся только метаданные файлов.
7. Для временного доступа использовать signed URL с коротким сроком жизни.

## 3. Метаданные файла

Таблица `files` должна хранить:

- `id`;
- `original_name`;
- `storage_name`;
- `storage_path`;
- `disk`;
- `mime_type`;
- `size`;
- `hash`;
- `uploaded_by_user_id`;
- `related_type`;
- `related_id`;
- `status`;
- `created_at`;
- `updated_at`;
- `deleted_at`.

## 4. Связь файлов с сущностями

Файл может быть связан с:

- заявкой;
- стыком;
- заданием;
- результатом контроля;
- РК-снимком или пленкой;
- заключением;
- реестром;
- сменой;
- оборудованием;
- сотрудником;
- документом.

Если связь критична для предметной области, использовать отдельную pivot-таблицу или специализированную таблицу.

## 5. Документы

Система должна иметь единый реестр документов.

Документ хранит:

- тип;
- номер;
- дату;
- организацию;
- город;
- объект/участок;
- сотрудника;
- оборудование;
- заявку;
- заключение;
- срок действия;
- статус;
- файлы;
- версии;
- комментарий.

Не все поля обязательны для всех типов документов. Но нельзя добавлять поля без понимания, какой документ их использует.

## 6. Версии документов

Версионирование нужно для значимых документов:

- заключения;
- выданные PDF;
- документы с утверждением;
- документы, участвующие в архиве;
- документы, которые нельзя менять бесследно.

Версия должна фиксировать:

- номер версии;
- кто создал;
- когда создал;
- основание создания;
- файл версии;
- связанный документ;
- статус.

## 7. Заключения

Утвержденное заключение нельзя редактировать напрямую.

Исправления должны выполняться через:

- новую версию;
- замену;
- аннулирование;
- корректирующую запись.

Любое изменение результата контроля или заключения должно попадать в audit log.

## 8. Реестры и архив

Реестр фиксирует передачу материалов.

Типы реестров:

- передача пленок;
- передача заключений;
- передача в технический надзор;
- передача в ПТО;
- передача в архив;
- внутренние реестры лаборатории.

Архив связывает:

- заявку;
- стык;
- метод контроля;
- заключение;
- пленку/снимок;
- реестр;
- файлы.

## 9. Audit log

Audit log обязателен для:

- заявок;
- стыков;
- методов контроля;
- заданий;
- результатов;
- заключений;
- реестров;
- файлов;
- смен;
- оборудования;
- квалификаций;
- пользователей и прав.

Audit log должен хранить:

- кто изменил;
- когда изменил;
- какую сущность изменил;
- старые значения;
- новые значения;
- IP;
- user-agent;
- причину изменения, если она требуется;
- связанную операцию.

## 10. Критичные действия

Для критичных действий нужен audit log и, при необходимости, отдельное подтверждение:

- изменение результата контроля;
- утверждение заключения;
- аннулирование заключения;
- создание новой версии заключения;
- удаление файла;
- закрытие смены;
- списание оборудования;
- ручное изменение номера документа;
- изменение объекта/участка закрепления сотрудника;
- изменение роли пользователя.

## 11. File service

Работу с файлами следует централизовать в файловом сервисе.

Файловый сервис должен отвечать за:

- сохранение файла;
- расчет хеша;
- создание записи в БД;
- проверку прав скачивания;
- генерацию signed URL или stream-download;
- удаление/аннулирование файла;
- audit log критичных действий.

## 12. Запреты

Не делать:

- прямую отдачу файлов из `public`;
- хранение бинарных файлов в MySQL;
- удаление файла без audit log, если файл связан с производственными данными;
- редактирование утвержденного заключения без версии;
- перезапись файла версии без создания новой версии;
- использование публичных URL для приватных документов.
