# Проектный документ веб-приложения для учета неразрушающего контроля сварных соединений

## 1. Назначение системы

Веб-приложение предназначено для цифрового учета работ по неразрушающему контролю сварных соединений, ведения заявок, стыков, методов контроля, результатов, заключений, сменных журналов, реестров передачи, архива материалов, оборудования, сотрудников и связанных документов.

Система должна заменить ручной табличный учет и стать единой рабочей средой для участка или лаборатории НК. Основной фокус — простая, быстрая и надежная работа с производственными данными: заявками, сварными стыками, заданиями, результатами контроля, пленками, заключениями и сменными операциями.

Система проектируется сразу как production-level приложение: с ролевым доступом, аудитом, журналированием действий, файловым архивом, резервным копированием, очередями фоновых задач, контролем статусов и понятной структурой кода.

## 2. Главные принципы проектирования

Система должна быть простой в использовании, простой в сопровождении и достаточно строгой по данным.

Главные принципы:

1. Одна операция пользователя должна соответствовать реальному производственному действию.
2. Интерфейсы должны быть рабочими, плотными и быстрыми, без декоративной сложности.
3. База данных должна быть нормализованной, но без чрезмерной абстракции.
4. Все ключевые сущности должны иметь понятные таблицы, связи и статусы.
5. Результаты контроля, номера заключений, реестры и архивные записи нельзя менять бесследно.
6. Система должна хранить не только итог “годен / не годен”, но и контекст контроля.
7. Код должен быть модульным, типизированным, документированным и понятным для дальнейшей доработки.
8. Расчеты, статусы и переходы должны находиться в сервисном слое, а не размазываться по контроллерам и frontend.
9. Веб-приложение должно оставаться основным рабочим местом, а мобильные Android-решения — закрывать короткие полевые и сменные сценарии.
10. Архитектура должна быть рассчитана на solo-разработку и дальнейшее развитие без отдельной frontend-команды.

## 3. Предлагаемый технический стек

### 3.1 Backend

Основной backend:

* PHP 8.4;
* Laravel 12;
* MySQL 8.4;
* Redis;
* Laravel Queue;
* Laravel Scheduler;
* Supervisor для воркеров;
* Spatie Laravel Permission для ролей и прав;
* Spatie Laravel Activitylog или собственный audit log для журнала действий;
* Laravel Sanctum для API-токенов и мобильных/PWA-клиентов;
* PHPStan или Larastan для статического анализа;
* PHPUnit / Pest для тестов.

Laravel подходит как основной каркас, потому что система является типичным модульным монолитом: много форм, таблиц, прав, документов, статусов, файлов и связанных бизнес-процессов. Микросервисы здесь не нужны.

### 3.2 Frontend веб-приложения

Рекомендуемый вариант для простоты:

* Laravel Blade;
* Livewire 3;
* Alpine.js;
* Vite;
* Tailwind CSS;
* минимальный TypeScript только там, где он реально нужен.

Причина выбора: интерфейсы системы в основном состоят из таблиц, фильтров, форм, карточек, модалок, журналов и экранов согласования. Для этого не требуется тяжелая SPA-архитектура. Blade + Livewire позволяют быстрее разрабатывать production-интерфейсы одному PHP-разработчику и держать меньше frontend-кода.

Vue 3 можно использовать точечно для сложных интерактивных компонентов, если они появятся: визуальная схема стыков, продвинутый просмотрщик снимков, drag-and-drop распределение заданий, сложная таблица с массовым редактированием. Но базовая система не должна зависеть от Vue как обязательного слоя.

### 3.3 Мобильный слой

Основной вариант:

* PWA для Android-планшетов и смартфонов;
* адаптивные мобильные экраны;
* установка на главный экран Android;
* работа через API Laravel Sanctum;
* ограниченный offline-режим для черновиков и очереди отправки, если это потребуется для площадок с плохой связью.

Нативное Android-приложение на Kotlin не нужно закладывать как обязательное решение. Оно потребуется только при подтвержденной необходимости глубокой работы с устройством: Bluetooth/USB-оборудование, специализированные сканеры, тяжелый offline-режим, локальное хранение большого архива снимков или интеграция с аппаратурой.

### 3.4 Файлы, документы и экспорт

Для файлов и документов:

* приватное файловое хранилище Laravel Storage;
* S3-compatible хранилище;
* signed download URL;
* контроль прав при скачивании;
* хранение метаданных файлов в БД;
* версионирование значимых документов.

Для Excel:

* PhpSpreadsheet или Laravel Excel.

Для PDF:

* mPDF для русскоязычных печатных форм, таблиц и документов;
* HTML-шаблоны печатных форм;
* хранение сгенерированных PDF как версий документов, если документ был выдан или подписан.

### 3.5 Инфраструктура

Production-инфраструктура:

* Ubuntu Server LTS;
* Nginx;
* PHP-FPM;
* MySQL;
* Redis;
* Supervisor;
* cron для Laravel Scheduler;
* Docker Compose для разработки;
* отдельные backup-скрипты для БД и файлов;
* логирование Laravel logs + системные логи;
* мониторинг места на диске, очередей, ошибок и резервных копий.

Octane, Swoole, сложный Kubernetes и микросервисы не нужны. Для этой системы важнее предсказуемость, простое сопровождение и надежные резервные копии.

## 4. Архитектурный тип системы

Система должна быть модульным монолитом.

Основные backend-модули:

1. Auth — вход, пользователи, сессии, API-токены.
2. Access — роли, права, политики доступа.
3. Employees — сотрудники, должности, квалификации, удостоверения.
4. Organizations — организации, заказчики, лаборатория, реквизиты.
5. Objects — города, объекты/участки, титулы, чертежи, линии.
6. Welds — сварные соединения, сварщики, материалы, параметры стыков.
7. NdtRequests — заявки НК.
8. NdtTasks — задания исполнителям.
9. NdtResults — результаты контроля по методам.
10. Radiography — РК, пленки, снимки, пересветы, плотности, архив.
11. VisualControl — ВИК.
12. PenetrantControl — ПВК.
13. MagneticControl — МК.
14. UltrasonicControl — УК.
15. Conclusions — заключения, позиции заключений, статусы выдачи.
16. Registers — реестры передачи, акты, архивные передачи.
17. Shifts — смены лаборанта и дешифровщика.
18. Inventory — пленка, химия, расходники.
19. Equipment — оборудование, поверки, калибровки, ремонты, выдачи.
20. Documents — документы, файлы, версии, связи.
21. Notifications — уведомления, очередь отправки.
22. Audit — журнал действий и изменений.
23. Reports — отчеты, выгрузки, печатные формы.
24. Admin — системные справочники и настройки.

Каждый модуль должен иметь собственные модели, DTO, FormRequest-классы, сервисы, политики, маршруты и тесты. Контроллеры должны быть тонкими.

## 5. Основные пользователи и роли

### 5.1 Администратор системы

Администратор видит все данные системы и управляет всеми разделами, пользователями, ролями, правами, справочниками, объектами, настройками и служебными операциями.

Может:

* видеть все объекты/участки и все данные по ним;
* создавать и блокировать пользователей;
* назначать роли;
* закреплять сотрудников за объектами/участками;
* управлять городами, объектами/участками и справочниками;
* управлять оборудованием и документами;
* видеть audit log;
* настраивать системные параметры;
* управлять файлами и архивом на уровне администратора;
* запускать служебные операции.

Администратор не должен бесследно менять производственные результаты контроля.

### 5.2 Начальник участка

Участок в системе — это объект в определенном городе. В одном городе может быть один или несколько объектов/участков. Каждый сотрудник закрепляется за конкретным объектом/участком.

Начальник участка отвечает за организацию работ НК на своем объекте/участке, принятие заявок, планирование, распределение заданий, контроль выполнения, утверждение результатов и выдачу заключений в рамках закрепленного объекта/участка.

Может:

* видеть данные своего объекта/участка;
* видеть сотрудников, закрепленных за своим объектом/участком;
* принимать заявки в работу по своему объекту/участку;
* назначать методы контроля;
* распределять задания исполнителям;
* контролировать статус заявок и стыков;
* утверждать или возвращать результаты;
* утверждать заключения;
* видеть отчеты по своему объекту/участку;
* контролировать оборудование, квалификации и просроченные документы по своему объекту/участку.

### 5.3 Инженер НК / Дешифровщик

Отвечает за анализ материалов контроля, дешифровку снимков/пленок, фиксацию дефектов, брака и подозрений на подлог, оформление результатов контроля, подготовку заключений и работу в сменном сценарии дешифровки.

Может:

* просматривать назначенные материалы контроля;
* выполнять дешифровку снимков/пленок;
* фиксировать просмотренные группы пленок/снимков;
* фиксировать дефекты;
* фиксировать брак;
* фиксировать подозрение на подлог;
* фиксировать очистку рабочего места в рамках смены дешифровщика;
* оформлять результаты контроля;
* формировать проекты заключений;
* отправлять заключения на утверждение;
* возвращать материалы на повторный контроль;
* завершать смену дешифровщика при наличии соответствующих прав.

### 5.4 Дефектоскопист

Отвечает за выполнение контроля на объекте или участке.

Может:

* видеть назначенные задания;
* видеть список стыков;
* отмечать доступность стыка;
* фиксировать выполнение контроля;
* указывать дату, место, метод, оборудование;
* прикреплять фото, файлы, снимки или служебные материалы;
* передавать результаты лаборанту или инженеру НК / дешифровщику.

### 5.5 Лаборант

Отвечает за сменные операции, прием материалов, работу с пленкой, химией, расходниками, подготовку и передачу материалов на дальнейшую обработку.

Может:

* начинать и завершать смену;
* принимать пленку и материалы;
* выдавать пленку;
* вести поступление и списание пленки;
* вести поступление, замену и списание химии;
* создавать запросы на химию;
* фиксировать регламентные работы;
* передавать материалы на дешифровку;
* формировать сменный отчет.

* объект / участок;
* титул;
* ответственное подразделение;
* приоритет;
* срок выполнения;
* основание работ;
* комментарий;
* приложенные файлы;
* список стыков;
* требуемые методы контроля.

Статусы заявки:

* черновик;
* зарегистрирована;
* на уточнении;
* принята в работу;
* запланирована;
* в работе;
* частично выполнена;
* ожидает анализа;
* на утверждении;
* выдана;
* закрыта;
* отменена.

### 6.2 Сварной стык

Стык является центральной производственной сущностью.

Карточка стыка должна хранить:

* город;
* объект / участок;
* титул;
* чертеж;
* лист;
* ревизию;
* линию;
* номер стыка;
* диаметр;
* толщину;
* материал 1;
* материал 2;
* дату сварки;
* сварщиков и их клейма;
* способ сварки;
* тип шва;
* категорию трубопровода;
* среду;
* PWHT;
* НТД / норму оценки;
* требуемые методы контроля;
* текущий статус;
* историю изменений.

Статусы стыка:

* создан;
* ожидает контроля;
* назначен в задание;

* контроль выполнен;
* ожидает анализа;

* дефект;
* требуется ремонт;
* в ремонте;
* ожидает повторного контроля;
* исключен;
* закрыт.

### 6.3 Методы контроля

Один стык может иметь несколько методов контроля.

Базовые методы:

* РК — радиографический контроль;
* ВИК — визуальный и измерительный контроль;
* ПВК — капиллярный контроль;
* МК — магнитопорошковый контроль;
* УК — ультразвуковой контроль.

Метод контроля не должен храниться строкой через запятую. Метод должен быть отдельной сущностью, связанной со стыком через назначение метода.

Для каждого метода должны быть:

* статус;
* исполнитель;
* дата контроля;
* оборудование;
* НТД;
* результат;
* дефекты;
* файлы;
* связь с заключением;
* история изменений.

### 6.4 Задание исполнителю

Задание связывает заявку, стыки, метод контроля, исполнителя, срок и статус выполнения.

Основные данные задания:

* номер задания;
* заявка;
* город;
* объект / участок;
* метод контроля;
* исполнитель;
* список стыков;
* плановая дата;
* приоритет;
* комментарий начальника участка;
* статус.

Статусы задания:

* создано;
* назначено;
* принято исполнителем;
* в работе;
* выполнено;
* выполнено частично;
* возвращено;
* отменено.

### 6.5 Выполнение контроля

Исполнитель открывает задание, видит список стыков и фиксирует результат по каждому стыку.

Минимальные действия:

* отметить стык как доступный или недоступный;
* указать факт выполнения контроля;
* указать дату и место;
* указать примененное оборудование;
* добавить комментарий;
* прикрепить файлы;
* передать результат на следующий этап.

Для полевой работы должен быть отдельный мобильный экран с минимальным количеством полей.

### 6.6 Радиографический контроль

РК должен иметь отдельный расширенный контур, потому что он связан с пленкой, снимками, пересветами, плотностями, браком, дешифровкой и архивом.

Для РК нужно хранить:

* номер заключения;
* дату контроля;
* дату заключения;
* исполнителя;
* клеймо дефектоскописта;
* количество снимков;
* тип пленки;
* баркод;
* плотности;
* досветы;
* статус пересвета;
* причину пересвета;
* дефекты;
* итог;
* реестры передачи;
* архивное местоположение;
* связанные файлы.

Статусы РК:

* назначен;
* снят;
* передан лаборанту;
* в обработке;
* готов к дешифровке;
* на дешифровке;
* требуется пересвет;
* пересвет выполнен;
* дешифрован;
* передан на анализ;
* включен в заключение;
* архивирован.

## 7. Сменные процессы

### 7.1 Общая логика смены

Смена создается для конкретного сотрудника, даты, объекта/участка и роли смены.

Типы смен:

* смена лаборанта;
* смена дешифровщика.

Общие статусы смены:

* открыта;
* в работе;
* ожидает завершения;
* завершена;
* отменена.

Один сотрудник не должен иметь две открытые смены одного типа одновременно.

### 7.2 Смена лаборанта

Смена лаборанта нужна для учета операций с пленкой, химией, расходниками и регламентными работами.

Функции смены лаборанта:

* старт смены;
* прием пленки;
* выдача пленки;
* списание пленки;
* поступление химии;
* замена химии;
* запрос химии;
* закрытие запроса химии;
* регламентные работы;
* передача материалов на дешифровку;
* завершение смены;

## 17. Бизнес-правила

1. Пользователь со статусом “заблокирован” не может работать в системе.
2. Производственные действия разрешаются только пользователям с нужной ролью и правом.
3. Администратор видит все объекты/участки и управляет всеми разделами.
4. Начальник участка работает с данными своего объекта/участка.
5. Каждый сотрудник закреплен за конкретным объектом/участком.
6. Объект/участок относится к городу.
7. В одном городе может быть один или несколько объектов/участков.
8. Заявка не может перейти в работу без стыков.
9. Стык не может быть закрыт без результата по обязательным методам контроля.
10. Один стык может иметь несколько методов контроля.
11. Один метод контроля может иметь несколько попыток или повторных результатов, если был пересвет, ремонт или повторный контроль.
12. Утвержденное заключение нельзя редактировать напрямую.
13. Изменение результата контроля должно фиксироваться в аудите.
14. Номер заключения должен быть уникален в рамках заданного правила нумерации.
15. Оборудование без действующей поверки/калибровки не должно назначаться в результат контроля.
16. Исполнитель без действующей квалификации по методу не должен назначаться на соответствующий метод, если включен строгий контроль квалификаций.
17. Смена не может быть завершена без обязательных журналов по типу смены.
18. Файл нельзя скачать без права доступа к связанной сущности.
19. Справочник нельзя удалить, если он используется в рабочих данных.
20. Все массовые операции должны писать audit log.

* ручное изменение номера документа;
* изменение объекта/участка закрепления сотрудника.

## 20. API

API нужно проектировать для мобильных/PWA-клиентов и возможных интеграций.

Группы API:

* /api/auth;
* /api/profile;
* /api/mobile/tasks;
* /api/mobile/shifts;
* /api/mobile/welds;
* /api/mobile/files;
* /api/mobile/equipment;
* /api/notifications.

Внутренний веб-интерфейс может работать через Livewire, а мобильные сценарии — через JSON API.

API должно использовать:

* Sanctum;
* rate limiting;
* проверку прав через policies;
* DTO для входных данных;
* единый формат ошибок;
* логирование критичных действий.

## 21. Производительность

Основные требования:

* все списки с пагинацией;
* фильтры выполняются на стороне БД;
* индексы под реальные фильтры;
* тяжелые экспорты — через очередь;
* генерация PDF — через очередь;
* загрузка больших файлов — потоковая;
* изображения и документы не должны грузиться в память целиком без необходимости;
* dashboard не должен выполнять тяжелые агрегаты на каждом запросе;
* для счетчиков использовать кэш с контролируемым сбросом;
* не использовать N+1-запросы;
* на карточках использовать eager loading только нужных связей.

## 22. Безопасность

Обязательные меры:

* HTTPS;
* защищенные cookies;
* CSRF для веб-форм;
* rate limiting для login и API;
* RBAC;
* policies на каждую важную сущность;
* приватные файлы;
* signed URL с коротким сроком жизни;
* журналирование входов;
* блокировка пользователя;
* аудит прав;
* резервное копирование;
* запрет бесследного удаления критичных данных.

Удаление рабочих производственных данных должно быть мягким, через soft delete или статус “аннулировано”, если данные уже участвовали в документах.

## 23. Резервное копирование и эксплуатация

Production должен иметь:

* ежедневный backup MySQL;
* ежедневный backup файлов;
* хранение нескольких последних копий;
* проверку успешности backup;
* отдельный лог ошибок backup;
* инструкцию восстановления;
* мониторинг свободного места;
* мониторинг очередей;
* мониторинг ошибок Laravel;
* логирование фоновых задач;
* health-check endpoint.

Минимальные сервисы на сервере:

* nginx;
* php-fpm;
* mysql;
* redis;
* supervisor;
* cron.

## 24. Стандарты кода

Backend-код должен быть строгим и понятным.

Требования:

* `declare(strict_types=1);`;
* типизированные свойства и методы;
* DTO для сложных входных данных;
* FormRequest для валидации;
* Service-классы для бизнес-операций;
* Repository использовать только там, где он реально упрощает код;
* Policies для доступа;
* Enum-классы для статусов и типов;
* PHPDoc для публичных методов сервисов, DTO, сложных моделей и нетривиальных расчетов;
* комментарии по ходу кода только там, где без них непонятна предметная логика;
* миграции с индексами и FK;
* сидеры для базовых справочников;
* тесты для критичных переходов статусов.

Контроллер не должен содержать бизнес-логику. Контроллер принимает запрос, вызывает сервис, возвращает ответ.

## 25. Тестирование

Обязательные тесты:

* авторизация;
* права доступа;
* создание города;
* создание объекта/участка;
* закрепление сотрудника за объектом/участком;
* ограничение начальника участка данными своего объекта/участка;
* полный доступ администратора ко всем объектам/участкам;
* создание заявки;
* добавление стыков;
* назначение методов;
* назначение задания;
* выполнение задания;
* создание результата;
* результат с дефектом;
* пересвет;
* формирование заключения;
* утверждение заключения;
* запрет редактирования утвержденного заключения;
* старт и завершение смены лаборанта;
* старт и завершение смены дешифровщика;
* движение пленки;
* движение химии;
* запрет назначения неисправного оборудования;
* запрет скачивания файла без прав;
* audit log критичных действий.

## 26. Критерии готовности production-версии

Система считается готовой к production, когда:

1. Все основные роли заведены и проверены.
2. Администратор видит все данные и управляет всеми разделами.
3. Начальник участка видит и контролирует данные своего объекта/участка.
4. Каждый сотрудник закреплен за объектом/участком.
5. Города и объекты/участки заведены как отдельные сущности.
6. Заявки, стыки, методы, задания, результаты и заключения работают как единый процесс.
7. Смены лаборанта и дешифровщика имеют обязательные журналы и отчеты.
8. РК, ВИК, ПВК, МК и УК поддерживаются как отдельные методы контроля.
9. Реестры, акты и архив связаны с результатами и заключениями.
10. Файлы хранятся приватно и скачиваются только с проверкой прав.
11. Все критичные изменения пишутся в audit log.
12. Работают очереди для тяжелых операций.
13. Настроены backup БД и файлов.
14. Настроены роли, права, политики и блокировка пользователей.
15. Утвержденные заключения защищены от прямого изменения.
16. Списки работают быстро на больших объемах данных.
17. Есть базовые автотесты критичных бизнес-сценариев.
18. Код документирован PHPDoc там, где это нужно для понимания предметной логики.
19. Есть эксплуатационная инструкция для запуска, обновления, backup и восстановления.

## 27. Итоговая архитектурная рекомендация

Оптимальная архитектура — Laravel-монолит с MySQL, Redis, очередями, приватным файловым хранилищем, RBAC, audit log, Blade/Livewire-интерфейсом и отдельными PWA-экранами для Android.

Такой подход дает баланс:

* достаточно простой код;
* понятная БД;
* высокая скорость разработки;
* нормальная производительность;
* удобная эксплуатация;
* возможность мобильной работы без отдельного тяжелого Android-проекта;
* возможность дальнейшего расширения без перестройки архитектуры.
