# Стратегия тестирования

## 1. Назначение

Документ фиксирует обязательные направления тестирования. Codex должен добавлять или обновлять тесты при изменении бизнес-логики, статусов, прав, файлов, смен и заключений.

## 2. Базовый стек тестирования

Использовать:

- PHPUnit или Pest;
- Laravel Feature tests;
- Laravel Unit tests;
- factories;
- seeders для базовых ролей, permissions и справочников;
- fake storage для файлов;
- fake queue для очередей;
- fake notifications для уведомлений.

## 3. Общие правила

1. Критичная бизнес-логика должна иметь тесты.
2. Тесты должны проверять не только успешный сценарий, но и запреты.
3. Любая проверка доступа должна покрываться тестом хотя бы на уровне feature.
4. Изменение статусов должно покрываться тестами.
5. Работа с файлами должна проверять права доступа.
6. Утвержденные заключения должны быть защищены тестами от прямого редактирования.

## 4. Авторизация и пользователи

Тесты:

- пользователь может войти;
- заблокированный пользователь не может работать в системе;
- пользователь без роли не получает доступ к закрытым разделам;
- администратор видит все объекты/участки;
- начальник участка видит только свой объект/участок;
- смена роли пользователя пишется в audit log;
- блокировка пользователя пишется в audit log.

## 5. Города, объекты/участки и сотрудники

Тесты:

- создание города;
- создание объекта/участка;
- объект/участок относится к городу;
- сотрудник закрепляется за объектом/участком;
- изменение объекта/участка сотрудника пишется в audit log;
- начальник участка не видит чужой объект/участок;
- администратор видит все объекты/участки.

## 6. Заявки

Тесты:

- создание заявки;
- заявка требует объект/участок;
- добавление стыков в заявку;
- заявка не может перейти в работу без стыков;
- заявка переходит в статус `принята в работу`;
- история статусов заявки сохраняется;
- действия с заявкой пишутся в audit log, если они критичны.

## 7. Стыки

Тесты:

- создание стыка;
- стык относится к объекту/участку;
- стык может иметь несколько методов контроля;
- методы контроля не хранятся строкой;
- изменение статуса стыка сохраняется в истории;
- стык с дефектом может перейти в ремонт;
- после ремонта стык может перейти на повторный контроль.

## 8. Задания

Тесты:

- создание задания;
- назначение исполнителя;
- исполнитель видит свое задание;
- другой исполнитель не видит чужое задание;
- начальник участка видит задания своего объекта/участка;
- задание можно принять в работу;
- задание можно завершить;
- завершение задания меняет статус;
- отмена задания требует права.

## 9. Результаты контроля

Тесты:

- создание результата;
- результат связан со стыком и методом контроля;
- результат может содержать дефекты;
- изменение результата пишется в audit log;
- результат с дефектом меняет дальнейший статус стыка;
- результат можно передать на анализ;
- результат можно включить в заключение.

## 10. Радиографический контроль

Тесты:

- создание РК-результата;
- добавление пленки;
- добавление снимка;
- фиксация плотности;
- фиксация пересвета;
- пересвет переводит результат в нужный статус;
- после пересвета результат снова может перейти на дешифровку;
- архивная запись связывается с результатом и заключением.

## 11. Смена лаборанта

Тесты:

- старт смены лаборанта;
- нельзя открыть две смены лаборанта одновременно для одного сотрудника;
- прием пленки;
- выдача пленки;
- списание пленки;
- запрос химии;
- поступление химии;
- замена химии;
- фиксация регламентных работ;
- завершение смены;
- смена не завершается без обязательных данных.

## 12. Смена дешифровщика

Тесты:

- старт смены дешифровщика;
- нельзя открыть две смены дешифровщика одновременно для одного сотрудника;
- фиксация просмотренных групп пленок/снимков;
- фиксация брака;
- фиксация подозрения на подлог;
- фиксация очистки рабочего места;
- внесение результата дешифровки;
- передача результата на анализ;
- завершение смены;
- смена не завершается без обязательных данных.

## 13. Заключения

Тесты:

- создание проекта заключения;
- добавление результатов в заключение;
- отправка заключения на утверждение;
- утверждение заключения;
- возврат заключения на доработку;
- утвержденное заключение нельзя редактировать напрямую;
- создание новой версии заключения;
- аннулирование заключения;
- изменение заключения пишется в audit log.

## 14. Файлы

Тесты:

- загрузка файла;
- файл сохраняется в private storage;
- метаданные файла сохраняются в БД;
- пользователь с правом может скачать файл;
- пользователь без права не может скачать файл;
- прямой storage path не возвращается;
- удаление/аннулирование файла пишется в audit log.

## 15. Оборудование

Тесты:

- создание оборудования;
- фиксация поверки;
- фиксация калибровки;
- фиксация ремонта;
- оборудование без действующей поверки/калибровки не назначается в результат контроля, если включена строгая проверка;
- списание оборудования пишется в audit log.

## 16. Очереди и фоновые задачи

Тесты:

- тяжелый экспорт ставится в очередь;
- PDF-генерация ставится в очередь;
- уведомление ставится в очередь или отправляется через notification service;
- ошибка фоновой задачи логируется.

## 17. Минимальный набор тестов для новой функции

При реализации новой функции Codex должен проверить:

1. Есть ли права доступа.
2. Есть ли валидация входных данных.
3. Есть ли изменение статуса.
4. Нужен ли audit log.
5. Нужны ли файлы.
6. Нужны ли уведомления.
7. Нужны ли тесты на запреты.
