# Бизнес-процессы

## 1. Назначение

Документ фиксирует основные workflows. Codex должен использовать его при реализации сервисов, статусов, проверок, тестов и UI-действий.

## 2. Общий производственный процесс

```text
Заявка
→ регистрация
→ проверка полноты данных
→ список стыков
→ назначение методов НК
→ планирование заданий
→ выполнение контроля
→ передача материалов
→ лабораторная обработка / дешифровка
→ анализ результата
→ заключение
→ утверждение
→ выдача
→ архив
```

## 3. Заявка

### 3.1 Создание заявки

Сценарий:

1. Пользователь создает заявку.
2. Начальник участка открывает отдельную страницу создания или импорта заявки.
3. Объект/участок и заказчик подставляются автоматически по закреплению начальника участка.
4. Указывает номер, дату, титул, срок, комментарий и список стыков либо загружает CSV/XLSX.
5. Стыки автоматически регистрируются в системе.
6. Заявка получает статус `черновик` или `зарегистрирована`.

### 3.2 Принятие заявки в работу

Сценарий:

1. Начальник участка проверяет заявку.
2. Проверяет наличие стыков.
3. Проверяет методы контроля.
4. Переводит заявку в работу.

Правило:

- заявка не может перейти в работу без стыков.

## 4. Стык

### 4.1 Создание стыка

Стык создается в рамках заявки или как отдельная производственная запись, если такой сценарий будет реализован.

Минимальные данные:

- город;
- объект/участок;
- титул;
- чертеж;
- линия;
- номер стыка;
- диаметр;
- толщина;
- материалы;
- сварщики;
- НТД;
- требуемые методы контроля.

### 4.2 Жизненный цикл стыка

```text
создан
→ ожидает контроля
→ назначен в задание
→ в контроле
→ контроль выполнен
→ ожидает анализа
→ годен / дефект / требуется ремонт
→ закрыт
```

При ремонте:

```text
дефект
→ требуется ремонт
→ в ремонте
→ ожидает повторного контроля
→ назначен в задание
→ контроль выполнен
```

## 5. Назначение задания

Сценарий:

1. Начальник участка выбирает заявку, метод контроля и стыки.
2. Назначает исполнителя.
3. Указывает плановую дату и комментарий.
4. Система создает задание.
5. Исполнитель получает уведомление.

Правила:

- задание должно относиться к объекту/участку;
- исполнитель должен быть доступен по роли и объекту/участку;
- оборудование без действующей поверки/калибровки не должно назначаться в результат контроля, если включена строгая проверка.

## 6. Выполнение контроля

Сценарий:

1. Исполнитель открывает назначенное задание.
2. Видит список стыков.
3. По каждому стыку фиксирует выполнение.
4. Указывает дату, метод, оборудование, результат и комментарий.
5. Прикрепляет файлы, если нужно.
6. Передает результат на следующий этап.

## 7. Радиографический контроль

РК имеет расширенный процесс.

```text
назначен
→ снят
→ передан лаборанту
→ в обработке
→ готов к дешифровке
→ на дешифровке
→ дешифрован / требуется пересвет
→ передан на анализ
→ включен в заключение
→ архивирован
```

При пересвете:

```text
требуется пересвет
→ пересвет выполнен
→ готов к дешифровке
→ на дешифровке
→ дешифрован
```

РК должен учитывать:

- пленки;
- снимки;
- количество снимков;
- тип пленки;
- баркод;
- плотности;
- досветы;
- пересветы;
- дефекты;
- итог;
- реестры передачи;
- архивное местоположение.

## 8. Смена лаборанта

### 8.1 Старт смены

Сценарий:

1. Лаборант открывает смену.
2. Система фиксирует сотрудника, объект/участок, тип смены и время старта.
3. Смена получает статус `открыта` или `в работе`.

Правило:

- один сотрудник не должен иметь две открытые смены лаборанта одновременно.

### 8.2 Операции смены

Смена лаборанта включает:

- прием пленки;
- выдачу пленки;
- списание пленки;
- поступление химии;
- замену химии;
- запрос химии;
- закрытие запроса химии;
- регламентные работы;
- передачу материалов на дешифровку.

### 8.3 Завершение смены

Обязательные данные завершения:

- дата и время завершения;
- выполненные регламентные работы;
- остатки / движения пленки;
- движения химии;
- незакрытые запросы;
- комментарий;
- подтверждение завершения.

## 9. Смена дешифровщика

### 9.1 Старт смены

Сценарий:

1. Инженер НК / Дешифровщик открывает смену дешифровщика.
2. Система фиксирует сотрудника, объект/участок, тип смены и время старта.
3. Смена получает статус `открыта` или `в работе`.

Правило:

- один сотрудник не должен иметь две открытые смены дешифровщика одновременно.

### 9.2 Операции смены

Смена дешифровщика включает:

- получение списка пленок/снимков;
- фиксацию просмотренных групп;
- фиксацию брака;
- фиксацию подозрения на подлог;
- фиксацию очистки рабочего места;
- внесение результатов дешифровки;
- передачу результатов на анализ.

### 9.3 Завершение смены

Обязательные данные завершения:

- журнал просмотренной пленки;
- отметка об очистке рабочего места;
- сведения о браке;
- сведения о подозрениях на подлог;
- список дешифрованных материалов;
- комментарий;
- подтверждение завершения.

## 10. Заключение

### 10.1 Подготовка заключения

Сценарий:

1. Инженер НК / Дешифровщик выбирает результаты, готовые к заключению.
2. Формирует проект заключения.
3. Проверяет номер, дату, метод, объект/участок, список стыков, НТД и итог.
4. Отправляет заключение на утверждение.

### 10.2 Утверждение заключения

Сценарий:

1. Начальник участка проверяет заключение.
2. Утверждает или возвращает на доработку.
3. После утверждения заключение защищается от прямого редактирования.

Правило:

- утвержденное заключение нельзя редактировать напрямую.

### 10.3 Исправление заключения

Исправления возможны через:

- новую версию;
- замену;
- аннулирование;
- корректирующую запись.

## 11. Реестр и архив

Сценарий:

1. Пользователь формирует реестр передачи.
2. Добавляет позиции: пленки, заключения, документы или архивные материалы.
3. Указывает отправителя, получателя, дату и объект/участок.
4. Переводит реестр в статус `сформирован` или `передан`.
5. После принятия реестр получает статус `принят` или `закрыт`.

Архив связывает заявку, стык, метод контроля, заключение, пленку/снимок, реестр и файлы.
