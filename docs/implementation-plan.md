# Операционный план реализации проекта

Документ составлен по `docs/project.md` как основному источнику требований. Остальные документы из `docs` использованы как уточняющие материалы, если они согласуются с `docs/project.md`. `docs/legacy` не используется как источник требований.

## 1. Основа реализации

Нужно реализовать production-level веб-приложение для учета неразрушающего контроля сварных соединений.

Тип системы: модульный монолит на Laravel.

Основные роли:

- Администратор системы;
- Начальник участка;
- Инженер НК / Дешифровщик;
- Дефектоскопист;
- Лаборант.

Основные процессы:

- регистрация заявки;
- ведение стыков;
- назначение методов НК;
- планирование и выполнение заданий;
- фиксация результатов контроля;
- расширенный контур РК;
- смена лаборанта;
- смена дешифровщика;
- формирование, утверждение, выдача и версионирование заключений;
- формирование реестров, актов и архива;
- учет оборудования, поверок, калибровок, ремонтов и выдач;
- учет сотрудников, квалификаций и документов;
- работа с приватными файлами;
- уведомления;
- audit log критичных действий.

Основные модули:

- `Auth`;
- `Access`;
- `Employees`;
- `Organizations`;
- `Objects`;
- `Welds`;
- `NdtRequests`;
- `NdtTasks`;
- `NdtResults`;
- `Radiography`;
- `VisualControl`;
- `PenetrantControl`;
- `MagneticControl`;
- `UltrasonicControl`;
- `Conclusions`;
- `Registers`;
- `Shifts`;
- `Inventory`;
- `Equipment`;
- `Documents`;
- `Notifications`;
- `Audit`;
- `Reports`;
- `Admin`.

В реализацию не входят:

- код нативных Android-приложений;
- микросервисы;
- Kubernetes;
- Octane;
- Swoole;
- обязательный Vue frontend;
- тяжелая SPA-архитектура;
- GraphQL;
- Elasticsearch;
- MongoDB.

PWA-экраны для Android-планшетов и смартфонов входят в план как часть Laravel/Blade/Livewire/API-реализации. Код Android-приложений в план не включается.

## 2. Рабочая декомпозиция проекта

Backend:

- Laravel 12;
- модульная структура `app/Modules`;
- тонкие контроллеры;
- FormRequest для HTTP-валидации;
- DTO для сложных входных данных;
- сервисы для бизнес-операций;
- policies/gates для доступа;
- enum-классы для статусов и типов;
- jobs для тяжелых операций.

База данных:

- MySQL 8.4;
- нормализованные таблицы по модулям;
- FK для обязательных связей;
- индексы под реальные фильтры;
- soft delete или статус аннулирования для рабочих сущностей;
- сидеры ролей, прав и базовых справочников.

API:

- Laravel Sanctum;
- `/api/auth`;
- `/api/profile`;
- `/api/mobile/tasks`;
- `/api/mobile/shifts`;
- `/api/mobile/welds`;
- `/api/mobile/files`;
- `/api/mobile/equipment`;
- `/api/notifications`;
- единый формат ответов и ошибок;
- rate limiting;
- проверка прав через policies.

Авторизация и роли:

- Spatie Laravel Permission;
- пять базовых ролей;
- permissions в формате `<resource>.<action>`;
- объектные ограничения по объекту/участку;
- блокировка пользователя;
- audit log для изменений ролей и доступа.

Административный интерфейс:

- пользователи;
- роли и права;
- города;
- объекты/участки;
- справочники;
- системные настройки;
- audit log;
- служебные операции.

Пользовательские интерфейсы:

- Рабочий стол;
- Заявки;
- Стыки;
- Задания;
- Результаты контроля;
- РК;
- Заключения;
- Смены;
- Оборудование;
- Сотрудники;
- Документы и архив;
- Справочники;
- PWA-экраны для коротких рабочих сценариев.

Файловое хранилище:

- S3-compatible хранилище;
- метаданные файлов в БД;
- скачивание только через backend;

Отчеты и документы:

- PDF через mPDF;

- HTML-шаблоны печатных форм;
- версии значимых документов;

Интеграции:

## 3. Сущности и модули, которые нужно реализовать

### Auth

Назначение: вход, сессии, API-токены, блокировка пользователя.

Связи: пользователь связан с сотрудником и ролями.

Кто использует: все роли.

Где участвует: web UI, API, PWA, audit log.

Этапы: 1, 2, 3, 17.

### Access

Назначение: роли, permissions, policies, объектные ограничения.

Связи: пользователи, сотрудники, объекты/участки, все критичные сущности.

Кто использует: все роли.

Где участвует: все backend-операции, UI-действия, API.

Этапы: 2, 3, 17.

### Objects

Назначение: города, объекты/участки, титулы, чертежи, линии.

Связи: объект/участок относится к городу; сотрудники, заявки, стыки, оборудование и смены относятся к объекту/участку.

Кто использует: Администратор системы, Начальник участка, все производственные роли через фильтры.

Где участвует: объектные ограничения, заявки, стыки, задания, отчеты.

Этапы: 4, 17.

### Employees

Назначение: сотрудники, должности, квалификации, удостоверения.

Связи: сотрудник закреплен за объектом/участком, может быть связан с пользователем, заданиями, сменами, результатами и заключениями.

Кто использует: Администратор системы, Начальник участка, все роли как исполнители процессов.

Где участвует: доступ, назначения, смены, квалификации, предупреждения.

Этапы: 4, 8, 17.

### Organizations

Назначение: заказчики, лаборатория, реквизиты, контактные лица.

Связи: заявки и документы могут ссылаться на организацию.

Кто использует: Администратор системы, Начальник участка.

Где участвует: заявки, документы, заключения.

Этапы: 5, 12.

### Welds

Назначение: учет сварных стыков как центральной производственной сущности.

Связи: объект/участок, заявка, материалы, сварщики, методы контроля, задания, результаты, заключения, архив.

Кто использует: Начальник участка, Дефектоскопист, Инженер НК / Дешифровщик.

Где участвует: заявки, задания, результаты, заключения, реестры, архив.

Этапы: 5, 6, 7, 10, 11, 12.

### NdtRequests

Назначение: входящий объем работ.

Связи: организация, объект/участок, титул, стыки, требуемые методы контроля, задания, результаты, заключения, файлы.

Кто использует: Начальник участка, Администратор системы.

Где участвует: основной производственный процесс.

Этапы: 5, 6, 12, 13.

### NdtTasks

Назначение: назначение исполнителю выполнить контроль по методу на стыках.

Связи: заявка, объект/участок, метод контроля, исполнитель, стыки, результаты.

Кто использует: Начальник участка, Дефектоскопист, Инженер НК / Дешифровщик.

Где участвует: планирование, выполнение контроля, PWA “Мои задания НК”.

Этапы: 6, 13.

### NdtResults

Назначение: общий результат выполнения метода контроля по стыку.

Связи: стык, метод, задание, исполнитель, оборудование, НТД, дефекты, файлы, заключение.

Кто использует: Дефектоскопист, Инженер НК / Дешифровщик, Начальник участка.

Где участвует: анализ, заключения, отчеты, audit log.

Этапы: 7, 9, 10, 11, 12, 13.

### Radiography

Назначение: расширенный контур РК.

Связи: общий результат, пленки, снимки, экспозиции, пересветы, плотности, смены, реестры, архив.

Кто использует: Дефектоскопист, Лаборант, Инженер НК / Дешифровщик, Начальник участка.

Где участвует: РК-процесс, смены, заключения, архив.

Этапы: 8, 9, 10, 11, 12, 13.

### VisualControl, PenetrantControl, MagneticControl, UltrasonicControl

Назначение: метод-специфичные результаты ВИК, ПВК, МК, УК.

Связи: расширяют общий результат контроля.

Кто использует: Дефектоскопист, Инженер НК / Дешифровщик, Начальник участка.

Где участвует: результаты, заключения, реестры, акты.

Этапы: 7, 12, 13.

### Shifts

Назначение: смена лаборанта и смена дешифровщика.

Связи: сотрудник, объект/участок, РК-материалы, движения пленки и химии, дешифровка, отчеты смен.

Кто использует: Лаборант, Инженер НК / Дешифровщик.

Где участвует: сменные журналы, PWA-сценарии, audit log.

Этапы: 10, 13.

### Inventory

Назначение: пленка, химия, расходники и их движения.

Связи: смена лаборанта, РК-пленки, запросы химии.

Кто использует: Лаборант, Начальник участка.

Где участвует: смена лаборанта, отчеты, предупреждения.

Этапы: 10, 15.

### Equipment

Назначение: оборудование, поверки, калибровки, ремонты, выдачи, возвраты, перемещения, дефекты, списание.

Связи: объект/участок, документы, результаты контроля, задания.

Кто использует: Администратор системы, Начальник участка, исполнители результатов.

Где участвует: назначения, результаты, предупреждения, запрет неисправного оборудования.

Этапы: 8, 13, 15.

### Conclusions

Назначение: официальный итоговый документ по одному или нескольким результатам контроля.

Связи: метод, объект/участок, заявка, стыки, результаты, файлы, версии, документы.

Кто использует: Инженер НК / Дешифровщик, Начальник участка.

Где участвует: утверждение, выдача, архив, PDF.

Этапы: 11, 12, 13.

### Registers

Назначение: реестры передачи, акты, архивные передачи.

Связи: заключения, стыки, пленки/снимки, документы, файлы, архивные позиции.

Кто использует: Начальник участка, Инженер НК / Дешифровщик, Лаборант.

Где участвует: передача пленок, заключений, документов, архива.

Этапы: 12, 15.

### Documents

Назначение: единый реестр документов, файлов, версий и связей.

Связи: организации, города, объекты/участки, сотрудники, оборудование, заявки, заключения.

Кто использует: все роли по доступу.

Где участвует: файлы, версии, архив, удостоверения, заключения.

Этапы: 9, 11, 12, 15.

### Notifications

Назначение: системные уведомления о рабочих событиях.

Связи: пользователи, задания, заявки, результаты, заключения, смены, оборудование, квалификации.

Кто использует: все роли.

Где участвует: рабочий стол, API, фоновые задачи.

Этапы: 14, 15.

### Audit

Назначение: журнал критичных действий и изменений.

Связи: все производственные сущности, пользователи, права, файлы.

Кто использует: Администратор системы, расследование изменений.

Где участвует: все критичные операции.

Этапы: 3, 5-17.

### Reports

Назначение: отчеты, выгрузки, печатные формы.

Связи: заявки, стыки, результаты, заключения, смены, оборудование, документы.

Кто использует: Администратор системы, Начальник участка.

Где участвует: Excel, PDF, рабочий стол.

Этапы: 15.

### Admin

Назначение: системные справочники, настройки, служебные операции.

Связи: роли, права, справочники, settings, health-check.

Кто использует: Администратор системы.

Где участвует: настройка системы и сопровождение.

Этапы: 3, 4, 14, 16.

**Документы и отчёты:**

- Подготовить базовые policies и `before` для администратора.
- Добавить audit log для смены роли, блокировки и разблокировки пользователя.

- Этап 3 должен быть завершен перед критичными доменными workflow, чтобы изменения сразу писались в audit log.
- Этап 4 должен быть завершен перед заявками, стыками, оборудованием, сменами и объектными ограничениями.
- Этап 5 должен быть завершен перед методами НК, заданиями и результатами.
- Этап 6 должен быть завершен перед результатами контроля и PWA “Мои задания НК”.
- Этап 7 должен быть завершен перед заключениями, отчетами по результатам и частью API.
- Этап 8 должен быть завершен перед строгими проверками назначения оборудования и исполнителей.
- Этап 9 должен быть завершен перед полноценными заключениями, реестрами, архивом и скачиванием файлов через API.
- Этап 10 должен быть завершен перед заключениями по РК, сменными отчетами и архивом РК.
- Этап 11 должен быть завершен перед реестрами передачи заключений, PDF заключений и архивом.
- Этап 12 должен быть завершен перед финальной архивной проверкой и отчетами по реестрам.
- Этап 13 должен начинаться после готовности backend-сервисов заявок, заданий, результатов, смен, файлов и оборудования.
- Этап 14 зависит от готовых событий основных модулей.
- Этап 15 зависит от готовых данных заявок, стыков, результатов, заключений, смен, оборудования, документов и реестров.
- Этап 16 выполняется после настройки очередей, scheduler-задач, файлового хранилища и отчетов.
- Этап 17 выполняется только после завершения всех функциональных этапов.

Сущности, которые должны быть готовы перед API:

- пользователи, роли, permissions;
- сотрудники и объект/участок;
- заявки;
- стыки;
- методы НК;
- задания;
- результаты;
- РК;
- смены;
- оборудование;
- файлы;
- уведомления.

API, которые должны быть готовы перед PWA-интерфейсами:

- `/api/auth`;
- `/api/profile`;
- `/api/mobile/tasks`;
- `/api/mobile/shifts`;
- `/api/mobile/welds`;
- `/api/mobile/files`;
- `/api/mobile/equipment`;
- `/api/notifications`.

Данные, нужные для документов и отчетов:

- для заключений: результаты, стыки, метод НК, НТД, объект/участок, исполнитель, проверяющий, утверждающий, файлы;
- для реестров: тип, номер, дата, объект/участок, отправитель, получатель, позиции, заключения, стыки, файлы;
- для актов: данные реестра, связанные документы и позиции;
- для сменных отчетов: смена, сотрудник, объект/участок, движения пленки, движения химии, регламентные работы, дешифровка;
- для архива: заявка, стык, метод контроля, заключение, пленка/снимок, реестр, файлы;
- для оборудования: карточка оборудования, поверки, калибровки, ремонты, выдачи, документы.
